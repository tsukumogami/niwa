# Pragmatism Panel — DESIGN-niwa-mesh-reliability

## Findings

- **Hand-waving on the 9-vs-11 shirabe count gap (Verification Notes / Mitigations).** The Negative section accepts a documented "open item" and Phase 3 mitigation says "investigate the source ... and either produce full count parity or document the residual gap with concrete justification." This is exactly the "investigate during implementation" pattern flagged. **Fix:** bound the investigation now (name the suspected cause: shirabe-extensions resolution under multi `--add-dir`) and decide a stop condition — either parity or accept-and-document — before phase entry. **Advisory.**

- **Defense-in-depth `.gitignore` write is speculative.** Decision 1 removes `InstallChannelInfrastructure`'s per-repo skill write entirely (lines 347-359). With the source eliminated, the new `.gitignore` entry guards against "any historical commits and any future regressions that re-introduce the per-repo write." There is no current writer. **Fix:** drop the gitignore.go extension; if a regression re-introduces the write, a test would catch it more directly than a gitignore. **Blocking** if it ships as new public-facing apply behavior; **Advisory** if treated as a one-line safety belt.

- **`source_state_at_fork` field is YAGNI.** The redelegate response includes `source_state_at_fork` so the caller "can decide whether parallel execution is safe." But the caller already chose to redelegate knowing the source's state from a prior `niwa_query_task`. The field is redundant with information the caller already has and only serves a hypothetical "fork-safety reflection" use case. **Fix:** omit the field; if a real consumer surfaces, it's a non-breaking add. **Advisory.**

- **`maybeRegisterCoordinator` added to four handlers when one would do.** Phase 1 wires registration into `handleDelegate`, `handleQueryTask`, `handleSendMessage`, and `handleListOutboundTasks`. The design rationale is "idempotent and cheap, so being generous is negligible." The canonical fan-out call is `handleDelegate`; once the coordinator delegates anything it's registered, and the others are reached only after that. **Fix:** add only `handleDelegate`, plus one of the polling calls for the no-delegation-yet edge. Drop the other two. **Advisory** (the cost is small but it's three call sites that exist for hypothetical paths).

- **Phase 6 "diff-vs-fixture CI check for `buildSkillContent`" is scope creep.** Issues #92/97/108-114 are about runtime correctness; the skill text fixture-diff CI infra is a new test mechanism not requested by any issue. **Fix:** rewrite the skill text in Phase 6 (in scope), defer the CI drift-check infra to a follow-on issue. **Advisory.**

- **Phase 5 is deferrable.** Phase 5 closes #113 (`required_skills`) and #114 (`niwa_redelegate`). The design itself acknowledges #113's value-prop drops to "catch typos / declarative intent" once Decision 1 lands; #114 is mechanically simple but its primary recovery use case (taskstore_lost) only exists once Decision 2 (Phase 4) lands. Phase 5 has no dependents in this design except Phase 6 (which itself is deferrable). **Fix:** consider splitting Phase 5 into a follow-on milestone, leaving this design with Phases 1–4 + a targeted skill update. **Advisory** — keeping it in is reasonable if shipping cohesion matters more than scope discipline.

- **`SOURCE_BODY_LOST` error code exists for a "rare corner case."** The design itself describes `taskstore_lost` with envelope.json missing as the rare sub-case of an already-rare class. Given `body_overrides` already exists on `niwa_redelegate`, a generic error ("source missing, supply body via body_overrides") is enough; a dedicated error code is new public surface for a path that may never fire in production. **Fix:** return a generic error from redelegate when envelope.json is missing; document `body_overrides` as the recovery path. **Advisory.**

- **`required_skills` rationale (4) is forward-looking, not load-bearing.** The four reasons listed for keeping #113 in scope: (1) typo catching, (2) intent declaration, (3) cross-workspace drift, (4) prep for future per-spawn customization. Reasons (2)–(4) are not load-bearing for any current issue; only (1) is concretely useful, and Decision 1 makes it observable at runtime anyway (failed skill invocations surface immediately). The "Future Path" section already covers (4). **Fix:** if Phase 5 stays, base it on the typo-catch value alone; remove the speculative-future framing from Decision 3's rationale. **Advisory.**

- **Daemon-side state.json writer is a real but justified surface expansion.** Flagged in design as Negative + Mitigation. Re-using the existing flock'd writer is the right call and the design commits to it. Not a finding — calling out only because it deserves a tight review at implementation time.

## Summary verdict

The design is appropriately scoped overall: the four decisions map cleanly onto the nine open issues, alternatives are recorded with rationale, and Phases 1, 2, and 4 are tight. The notable concerns are accreted edges around Phase 5/6 and Decision 1: a `.gitignore` write defending against a writer that no longer exists, a redelegate response field that no caller needs, expanded auto-registration trigger sites, and a fixture-diff CI mechanism that doesn't appear in any issue's acceptance criteria. None of these are architecturally damaging — most are one-line trims — but they should be removed before this becomes a milestone of issues.

## Goal

Bring the niwa-mesh skill text and `docs/guides/sessions.md` back into lockstep with the runtime that issues <<ISSUE:1>> through <<ISSUE:7>> deliver: rewrite `buildSkillContent` to remove dead vocabulary, add the actually-routed message types, document the worker config inheritance contract, document `taskstore_lost` recovery via `niwa_redelegate`, and document the new `daemon` sub-object on `niwa_list_sessions`.

## Context

Design: `docs/designs/current/DESIGN-niwa-mesh-reliability.md`

The contract audit recorded in `wip/research/explore_niwa-mesh-reliability_r1_lead-mesh-skill-contract.md` enumerated nine first-class runtime claims in the current niwa-mesh skill, six of which were broken by the open issues plus the message-vocabulary section (which advertises `question.ask`/`question.answer`/`status.update` types that no handler dispatches and omits the actually-routed `task.delegate`/`task.ask`).

Once issues 1-7 land, every claim the rewritten skill makes must be a runtime guarantee. The skill is generated by `buildSkillContent` (`internal/workspace/channels.go:682-833`), so the rewrite is one Go source change plus the regenerated SKILL.md output that ships when `niwa apply` runs.

`docs/guides/sessions.md` already documents some related contracts (`daemon_warning`, `branch_warning`, the "When the session daemon crashes" section). It needs additions for the new `daemon` sub-object on `niwa_list_sessions` (<<ISSUE:3>>), the `taskstore_lost` recovery flow via `niwa_redelegate` (<<ISSUE:5>>+<<ISSUE:7>>), and the worker config inheritance contract (<<ISSUE:4>>).

This issue closes the documentation tail of the cluster but doesn't close any individual issue from #92-#114 directly — every issue is closed by its corresponding implementation issue.

## Acceptance Criteria

- [ ] `buildSkillContent` is updated to:
  - Remove `question.ask`, `question.answer`, `status.update` from the message vocabulary section.
  - Add `task.delegate` and `task.ask` to the message vocabulary, alongside the existing `task.progress`, `task.completed`, `task.abandoned`, `task.cancelled`.
  - Replace the "Worker asks coordinator" pattern paragraph to describe the post-fix path: the worker calls `niwa_ask(to="coordinator")`, which routes via the `roleRoot` redirect to the main instance's coordinator inbox; the coordinator picks up the question via `niwa_check_messages` or `niwa_await_task` (`status:"question_pending"`); the coordinator answers via `niwa_finish_task(ask_task_id, outcome="completed", result=...)`. Drop the misleading "tight timeout (60-120s)" framing.
  - Replace the description of the spawn fallback ("...otherwise the daemon can spawn a worker...") with the post-PR-#93 contract: when no live coordinator is registered, `niwa_ask(to="coordinator")` returns `{status:"no_live_session"}` synchronously.
  - Add a paragraph documenting the `taskstore_lost` lifecycle: when a task's task store is lost (manual cleanup, partial workspace destroy, or fresh checkout), the daemon transitions the task to `state="abandoned"` with `reason="taskstore_lost"`. Recovery: call `niwa_redelegate(source_task_id, body_overrides=...)` to re-fire with the original body. No `dangling` state is exposed; `inbox/dangling/` is forensic-only.
  - Add a paragraph documenting the worker config inheritance contract: workers spawned by `niwa_delegate` inherit the workspace's full `.claude/` tree (settings, plugins, skills, hooks, CLAUDE.local.md) as a baseline — the same configuration a user would see by running `claude` directly in the role's repo. The carve-out for MCP server inheritance (`--strict-mcp-config`) stays.
  - Add `niwa_redelegate` to the documented coordinator API surface, including the `source_state_at_fork` response signal that distinguishes recovery flows from active forks.
  - Document the `MISSING_SKILLS` queue-time gate as a typo-catch and intent-declaration mechanism.
  - Cross-link to `docs/guides/sessions.md` for the canonical session-lifecycle reference.
- [ ] `docs/guides/sessions.md` gains:
  - A section documenting the `daemon` sub-object on `niwa_list_sessions` (`{alive, pid, started_at}`), including the contract that `status` is the lifecycle marker and `daemon.alive` is the runtime health probe. Reference issue #116 for the deferred fleet-observability fields.
  - A section documenting `niwa_create_session`'s synchronous spawn-failure contract: `DAEMON_SPAWN_TIMEOUT` error code, rollback behavior, and how to recover.
  - A section documenting `taskstore_lost` recovery via `niwa_redelegate`, parallel to the existing "When the session daemon crashes" section.
  - A section documenting the worker config inheritance contract (what workers inherit, what is scoped via `--strict-mcp-config`).
- [ ] `buildSkillContent`'s output passes `gofmt` and the existing channel infrastructure tests.
- [ ] After `niwa apply`, the regenerated `<instanceRoot>/.claude/skills/niwa-mesh/SKILL.md` contains all the contract changes above.
- [ ] No reference to `dangling` as a user-visible state remains in either the skill text or `docs/guides/sessions.md`. The filesystem subdir name is implementation detail.
- [ ] No reference to `question.ask`, `question.answer`, or `status.update` remains in the skill text.

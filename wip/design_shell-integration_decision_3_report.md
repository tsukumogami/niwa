### Decision: Shell function subcommand scope

**Context**

The shell function wrapper generated by `niwa init <shell>` intercepts certain subcommands to add cd behavior. The fundamental constraint is that a compiled binary can't change the parent shell's working directory -- the shell function must perform the actual `cd`. Issue #31 describes two related pain points: (1) after `niwa create`, users must manually cd into the new workspace, and (2) there's no way to jump to a specific repo within an existing workspace.

Of niwa's eight current commands (init, create, apply, status, destroy, reset, token, version), only `create` produces a new directory that users want to navigate to. None of the others have a clear cd target. The repo-navigation requirement from issue #31 is a distinct use case that no existing command addresses -- it requires a new command.

**Assumptions**

- The `go` command can reuse the workspace registry infrastructure already present in `apply.go` (`resolveRegistryScope`, `config.LoadGlobalConfig`, `LookupWorkspace`). If the registry doesn't track repo paths within instances, `go` will need workspace state (`LoadState`) to resolve repo directories.
- Two intercepted subcommands (`create` and `go`) represent the long-term steady state. No foreseeable command beyond these needs shell-level cd behavior.

**Chosen: Intercept `create` + new `go` command**

The shell function intercepts two subcommands: `create` (post-create navigation to the new instance) and a new `go` command (navigation to existing workspaces and repos within them). Both use the same cd directive protocol -- the binary outputs a path directive on a dedicated channel, and the shell function performs cd.

The `go` command syntax is `niwa go [workspace] [repo]`. Without arguments, it navigates to the current workspace root. With a workspace name, it resolves via the global registry. With both workspace and repo, it navigates to the repo directory within the instance. The binary handles all resolution logic; the shell function only reads the resulting path and calls cd.

The shell function dispatch looks like:

```bash
niwa() {
  case "$1" in
    create|go) # capture output, extract directive, cd if present
      ...
      ;;
    *) command niwa "$@" ;;
  esac
}
```

**Rationale**

Alt 2 (create + go) is the right scope because it fully addresses issue #31's two requirements without over-engineering. The marginal shell function complexity over intercepting only `create` is one additional string in a case pattern -- negligible. Meanwhile, the `go` command fills a real gap: there's currently no way to navigate between workspaces or jump to a repo within one. The registry infrastructure already exists in `apply.go`, so the Go implementation builds on proven code.

A generic protocol that intercepts all commands (Alt 3) introduces output-buffering risks for no practical benefit -- only two commands need cd behavior, and no foreseeable command will join them. Intercepting `apply` (Alt 4) doesn't work because apply can target multiple instances and the user is typically already inside one.

Shipping `create` alone (Alt 1) would be simpler, but it leaves half of issue #31 unresolved and would require shell function changes later anyway. Since both commands share the same directive protocol, designing them together produces a cleaner interface than bolting `go` on later.

**Alternatives Considered**

- **Intercept `create` only**: Addresses post-create navigation but leaves repo-level navigation unresolved. Would require revisiting the shell function when `go` is eventually added. Rejected because the incremental cost of including `go` now is small relative to the cost of redesigning later.
- **Generic directive protocol (intercept everything)**: Wraps all niwa invocations and checks for cd directives after every command. Future-proof but introduces output buffering that could break interactive output or progress displays. Over-engineered when only two commands need interception.
- **Intercept `create` + `apply`**: `apply` doesn't have a clear single navigation target -- it can apply to multiple instances, and users running apply from inside an instance are already there. Doesn't address repo-level navigation at all.

**Consequences**

- The shell function has a fixed, small dispatch table (create and go). This is easy to audit and test.
- A new `go` command must be designed and implemented as a Go subcommand, adding to niwa's command surface.
- The cd directive protocol must be designed to work identically for both create and go, preventing protocol divergence.
- If a third command ever needs cd behavior, the shell function needs a one-line change to the case pattern -- but this is unlikely given niwa's command inventory.

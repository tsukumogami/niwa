# Completeness Review

## Verdict: FAIL

The PRD has several gaps where requirements lack corresponding ACs, the rank-3-removal user story is missing, and `niwa migrate-source` error/edge-case coverage is incomplete enough that an implementer would have to guess.

## Issues Found

1. **R16 (no observable latency) has no acceptance criterion**: R16 promises "no observable latency over the pre-PRD code path on the happy path" but no AC verifies this. The Decision section reasons that R6's single-fetch contract makes this true by construction, but a verifiable AC is missing. Suggested fix: either delete R16 (treat as a Known Limitation / Decision narrative), or add an AC that asserts the fake-server records exactly one tarball request and zero Contents API calls (most of this is already covered by AC-P1; promote AC-P1 to verify R16 as well, or add an AC that no `/contents/` endpoint is hit).

2. **R17 (functional tests via `tarballFakeServer`) has no acceptance criterion**: R17 states the discovery code path "MUST be exercised by functional tests using the existing `tarballFakeServer` fixture." This is a meta-requirement about test coverage with no AC that verifies it. Suggested fix: either delete R17 as a Test Strategy note rather than a requirement, or add an AC that names a `test/functional/features/*.feature` file path that must exist and reference the fixture.

3. **No user story for the rank-3-removal regression case**: The review explicitly asked for this. AC-R1 covers the diagnostic, but no user story walks through "a developer with a brain repo that has only a root `niwa.toml` upgrades, hits this error, and resolves it" end-to-end. The Known Limitations section mentions it briefly. Suggested fix: add Story 8 "Rank-3 user upgrades and migrates" showing the user's first apply emitting the R4 diagnostic, the developer renaming `niwa.toml` to `.niwa/workspace.toml` (or adding the explicit-subpath flag), and re-running successfully. This makes the breaking change visible at the user-story level, not just at the AC level.

4. **`niwa migrate-source` without `--to` has incomplete failure-mode coverage (R8)**: R8 says "without `--to`, the command probes the current source for markers, infers the new slug if a single unambiguous migration target is found, and prints it for the user to confirm." But: (a) what happens when the probe finds NO marker at the current source? (b) what happens when the probe is ambiguous (multiple markers)? (c) what happens when the current `source_url` is unreachable on the network? AC-M4 only covers the happy case. Suggested fix: add R8 sub-clauses for these three failure modes (probably: reuse the R3/R4 diagnostics + a network warning), and add AC-M6 / AC-M7 / AC-M8 covering each.

5. **`niwa migrate-source --to <slug>` does not validate the destination slug (R8 / Q4)**: Q4 leans toward "accept any well-formed slug; let the next apply fail if discovery doesn't resolve." That leaning belongs in R8 as an explicit contract, not in an Open Question. As written, an implementer reading R8 could reasonably build either behavior. Suggested fix: lock Q4 into R8 ("the destination slug MUST be validated only for grammar (R1-R3 from upstream PRD); discovery resolution is deferred to the next `niwa apply`") and add an AC that `niwa migrate-source <name> --to "owner/repo:"` (empty subpath) fails at parse-time, while `niwa migrate-source <name> --to "owner/repo"` against a reachable but markerless source succeeds (the apply will fail later).

6. **`niwa migrate-source` against a workspace using the legacy working-tree shape is not addressed**: A user running this PRD's binary may still have a `<workspace>/.niwa/` that is a git working tree (per upstream R28's lazy conversion, which only fires on `apply`). What does `niwa migrate-source` do here? R9 says "MUST NOT delete the on-disk `.niwa/`" — so it leaves the working tree intact. Then `apply --force` per R10 has to handle BOTH a URL change AND a working-tree-to-snapshot upgrade in one pass. That combined path is not exercised by any AC. Suggested fix: add AC-M6 verifying that `niwa migrate-source` + `niwa apply --force` on a legacy working-tree workspace succeeds and converts atomically.

7. **R5 (on-disk byte-identical after discovery failure) is contradicted by AC-D5's "(or unchanged if init was aborted pre-Mkdir)"**: AC-D5 hedges: "The on-disk `<workspace>/` is removed (or unchanged if the init was aborted pre-Mkdir)." R5 is unconditional: "MUST leave the on-disk `<workspace>/.niwa/` byte-identical to its pre-init state. No partial snapshot is materialized." The hedge implies two possible end-states; R5 requires one. Suggested fix: tighten AC-D5 to assert the deterministic post-state — either "the workspace dir does not exist" (if init creates it lazily after discovery) or "the workspace dir exists but its `.niwa/` subdirectory does not exist" (if init creates the parent eagerly). Pick one and assert it consistently with whatever the design phase chooses.

8. **Upstream PRD reconciliation has no requirement and no AC**: The Out of Scope section says "Renaming or restructuring `PRD-workspace-config-sources.md`... is a process step left to the maintainer accepting this PRD." But the Goals section commits to "Close the R5 implementation gap" — which logically requires reconciling the upstream PRD's "Done" status. Either the reconciliation is a deliverable of this PRD (in which case it needs a requirement and an AC: e.g., "AC-U1: `docs/prds/PRD-workspace-config-sources.md` status field reads `In Progress` or carries an amendment block naming this PRD as the outstanding work"), or it is not (in which case the upstream PRD's status remains misleading). Suggested fix: add R18 explicitly committing to one of the two reconciliation options, and add a corresponding AC.

9. **R10 (no new behavior at apply time) has weak AC coverage**: R10 says this PRD "adds no new behaviour at apply time beyond what the upstream PRD already specifies" for the post-migrate-source apply path. AC-M3 verifies the happy path with `--force`, but there's no AC verifying the negative — that `niwa apply` WITHOUT `--force` after `migrate-source` still fails per upstream R26 (i.e., this PRD did not accidentally weaken the upstream gate). Suggested fix: add AC-M7: "After `niwa migrate-source my-workspace --to acme/vision` succeeds, `niwa apply my-workspace` (without `--force`) exits non-zero with stderr matching upstream R26's error contract."

10. **Backwards-compat AC-B1 conflates two concerns and skips one path**: AC-B1 says "the on-disk `.niwa/` (if it was a legacy working tree) is converted per upstream R28." But AC-B1's "given" doesn't actually establish whether `.niwa/` is a working tree or a snapshot; it just says "older binary." Older binaries always wrote working trees, but the AC's "if" hedge weakens it. Also, AC-B1 doesn't cover the case where the legacy working tree happens to be a `dot-niwa` repo with ONLY a root `workspace.toml` (the canonical rank-2 path this PRD promises preserves). Suggested fix: split AC-B1 into AC-B1a (legacy working tree → snapshot via rank-2, no prompt) and AC-B1b (registry-only upgrade with mirror-field write-on-first-save, no `.git/` present).

## Suggested Improvements

1. **Add a "Migration scenarios matrix" table to the user stories**: A small grid showing each starting state (registry shape × on-disk shape × source-repo shape) and the expected end state would help implementers and reviewers verify completeness at a glance. The current PRD requires the reader to assemble this mentally from R10-R12, AC-B1, AC-Y* (in upstream), and Stories 5-7.

2. **Cross-reference upstream ACs explicitly**: This PRD reuses many upstream concepts (R26 force-gate, R27 atomic swap, R28 lazy conversion, R12 atomic swap, R33 backwards compat). Several ACs depend on upstream ACs (AC-B1 leans on AC-Y4, AC-X1, AC-X3). A short "Inherited ACs" subsection that lists which upstream ACs are inherited verbatim, which are subsumed, and which are amended would close the "is this PRD self-contained or layered?" ambiguity flagged in Coverage Notes of the scope document.

3. **Pin the `niwa migrate-source` exit codes**: The command's exit codes are not specified. Tooling around the command (shell scripts, CI workflows) would benefit from explicit "exit 0 on success, exit 1 on registry not found, exit 2 on probe failure when --to is omitted" or similar. Defer the exact mapping to design, but commit to "exit codes are stable and documented in the help text" in R8.

4. **Clarify what `--yes` does (R8 / AC-M4)**: AC-M4 says `--yes` applies the rewrite immediately when discovery is unambiguous, but does it also bypass any confirmation when discovery is ambiguous? (Today's AC-M4 only covers the unambiguous case.) Clarify whether `--yes` is "accept the inferred slug" or "accept any inferred outcome including ambiguity resolution." Recommendation: `--yes` should only auto-confirm the unambiguous case; ambiguity should still fail.

## Summary

The PRD is structurally sound and the discovery/migrate-source split is well-reasoned, but it has six requirements (R5, R8, R10, R13, R16, R17) where the AC coverage is partial, weak, or missing entirely. The migrate-source command in particular has happy-path coverage but no specification for the four edge cases an implementer will inevitably hit (no marker at current source, ambiguous current source, network failure, legacy working-tree on disk). Tightening these gaps would let the next phase (design) proceed without backfilling requirements.

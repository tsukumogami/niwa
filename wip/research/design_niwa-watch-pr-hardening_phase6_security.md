# Security Review

## Verdict: PASS

The design extends state and adds a stop-and-resume continuation mechanism without opening a new egress channel or adding model trust; it re-asserts the shipped egress-denial containment on every resume and fails closed, and the residual findings are low-severity hardening notes for the plan rather than boundary weaknesses.

## Findings

1. **Resume-time OS sandbox re-assertion is asserted but load-bearing (not decomposed): Low** -> The Security section claims a resumed session is "no less caged than a fresh one" and that `ApplyReviewSettings` (sandbox stanza + hooks + post-guard) is re-asserted before `claude --resume`. This is the single most important claim in the design, and it rests on `claude --resume` launching through the *same* launch path that installs the OS no-egress sandbox over Bash ŌĆö not merely re-writing settings on disk. ED1's strongest control is the OS-level egress denial wrapping Bash; PreToolUse hooks and the post-guard are the second layer. If the resume launch path (generalized from `sessionattach/supervise.go`) re-applies hooks/settings but does not re-enter the OS sandbox wrapper that a fresh `dispatchLaunch` uses, containment silently degrades to the hook layer only. The doc treats "re-assert containment" as atomic; it should decompose into "the resume path invokes the same sandbox-applying launch as fresh dispatch." -> Mitigation: make the plan verify (and add an integration test asserting) that a *resumed* watch session's Bash is egress-denied at the OS layer, not just that hooks are present; state in the design that resume reuses the fresh-dispatch launch wrapper, not a lighter attach path.

2. **InstancePath and SessionID omitted from the explicit validate-before-use list: Low** -> Focus area 3 covers three id fields plus InstancePath. The Security section's validation bullet and the carried Mitigation ("validate all new persisted id fields") name only `ShortID` and `ConversationID` for `isSafeHandle` charset validation. `InstancePath` is the most path-traversal-relevant new field ŌĆö it is a checkout target and a path component for `instanceHasLiveJob` ŌĆö yet it is not in the validation list. It is niwa-generated and in scope at save time (not attacker-influenceable through PR content), so real risk is low, but the doc should state the confinement explicitly rather than leave the traversal-relevant field implicit. -> Mitigation: add "InstancePath must resolve under the managed instances root before use as a checkout/path target" to the validation list; note SessionID's use sites and whether it too reaches a CLI arg.

3. **Re-review prompt construction on Continue is unspecified: Low/Info** -> Component 6 delivers a "re-review prompt" on `claude --resume`. The doc does not say whether that prompt interpolates any PR-derived metadata (title, branch name, author). Egress denial bounds the blast radius regardless ŌĆö untrusted text into the model is not a new channel ŌĆö but PR-derived text must never be built into a shell command or CLI argument during resume. -> Mitigation: state that the re-review prompt is a fixed template and that PR content reaches the resumed model only as inert checkout data, never as command/argument text.

## Missing Considerations

Coverage is thorough. The one gap is inside finding 1: the Security section asserts resume-time containment as a single guarantee but does not break it into the testable sub-claim that matters (OS sandbox re-entry vs. hook-only re-assertion). Everything else the focus areas asked about is addressed in the doc: ancestry check on trusted GitHub API / trusted-CLI data rather than a git invocation inside the untrusted clone (no attacker-hook execution surface); the cap as a positive DoS/cost ceiling that Continue leaves cap-neutral (a request flood cannot exceed `maxStaged` because Continue reuses an already-counted agent and only Fresh consumes budget); determinism of the decision/freshness/cap as code over polled facts with no injectable model judge; and the compounded-context argument, which is valid ŌĆö a longer-lived, multi-diff transcript has no outbound channel, so it is the same sealed box for longer, not a new risk.

## Summary

This design preserves ED1's deterministic egress-denial boundary: the new continuation capability re-fetches the untrusted head as inert data via the shipped `FetchPRHead` path, re-asserts review settings before resume, fails closed if containment cannot be enforced, and keeps every decision (re-dispatch, freshness, cap) as model-free code. The three findings are low-severity plan-hardening notes ŌĆö chiefly that the "re-assert containment on resume" claim should be decomposed and tested as OS-sandbox re-entry, not just hook re-assertion, and that InstancePath should join the explicit validate-before-use list. None weakens the boundary or opens a new channel, so the verdict is PASS.

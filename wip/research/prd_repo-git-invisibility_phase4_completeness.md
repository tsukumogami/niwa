# Completeness Review

## Verdict: PASS

The PRD is buildable: every BRIEF scope item maps to a requirement, every functional requirement has at least one acceptance criterion, and the command surface (`niwa apply`, `niwa session create <repo> <purpose>`, `make test-functional`) and leak mechanisms (`.local`/`*.local*` convention, `.niwa/` worktree scaffold) all match the actual niwa codebase.

## Issues Found

1. **R3 names "re-sync" but no acceptance criterion exercises it.** R3 requires invisibility "after any re-sync of an existing worktree," and the Decisions section asserts re-sync is "covered transitively because it uses the same materialization path." But no AC runs a re-sync and asserts a clean `git status` — the AC only covers `session create`. The transitivity claim is an assumption, not a verified one. Suggested fix: either add an AC that re-syncs an existing worktree and asserts empty `git status --porcelain`, or have R7's functional test explicitly include a re-sync step in its worktree scenario so the "same materialization path" claim is tested rather than assumed.

2. **No requirement or AC covers the "warn when pattern missing" behavior the feature is replacing.** The problem statement says niwa today "only warns when the pattern is missing." The PRD makes invisibility self-guaranteed, which presumably makes that warning obsolete or wrong (it would now warn about a `.gitignore` niwa no longer depends on). The PRD is silent on what happens to that warning. Suggested fix: add a requirement stating the now-obsolete managed-repo `.gitignore` warning is removed or suppressed, with an AC asserting no such warning is emitted on apply. Otherwise an implementer may leave a stale, contradictory warning in place.

3. **"niwa-authored file" is the load-bearing term in R1/R3/R7 but is never defined.** The whole guarantee hinges on distinguishing niwa-authored output from user files, yet R7 deliberately forbids enumerating file names. The PRD does not state how the test fixture establishes the baseline (e.g., commit the fixture repo clean, then assert `git status --porcelain` is empty after niwa runs — anything niwa wrote that is not covered shows up). This is implied by "assert empty `git status`" but never made explicit as the operating definition. Suggested fix: add one sentence to R7 (or a definitions note) stating the test's notion of "niwa-authored leak" is "any working-tree change present after a niwa operation against a committed-clean fixture," which is exactly what makes the generic assertion work.

4. **R5 (preserve pre-existing coverage content) has an AC, but neither defines what "pre-existing content" can be.** R5 and its AC say niwa preserves "user content" already in the coverage location "without discarding or reordering." Since the coverage location is explicitly left to DESIGN (Out of Scope item 3), an implementer cannot yet know whether that location is a file niwa shares with the user (requiring careful append/merge) or a niwa-private file (where "user content" cannot exist). This is a latent contradiction: if the location is niwa-private and uncommitted (R2/R4), it is unclear how user content arrives there. Suggested fix: clarify whether R5 anticipates a shared location, or scope R5 to "niwa's own prior entries" (idempotent append) rather than "user content," and defer the shared-location case to DESIGN.

## Suggested Improvements

1. **Add an AC for the cross-platform / git-config interaction.** The likely mechanisms for uncommitted ignore coverage (`.git/info/exclude` or `core.excludesFile`) behave differently and can be overridden by user git config. An AC asserting the guarantee holds even when the user has a custom `core.excludesFile` set would harden R2/R4 against a real failure mode. Rationale: the coverage mechanism is the one part most likely to be silently defeated by environment, and the generic test won't catch it unless the fixture includes that condition.

2. **State the failure mode when niwa cannot record coverage.** Nothing says what happens if niwa fails to write its uncommitted coverage (read-only `.git`, permissions). Does apply fail loudly, or silently fall back to the old visible state? Rationale: silent fallback would reintroduce exactly the leak this feature removes, undetected. A one-line requirement (fail closed / surface an error) prevents that.

3. **Cross-reference the instance-root distinction explicitly in R2.** Verification confirmed the existing `EnsureInstanceGitignore` writes `*.local*` to the *instance root* `.gitignore`, which the PRD correctly puts out of scope — but R2 talks about "the managed repository's committed `.gitignore`," a different file. Adding a half-sentence noting that the instance-root mechanism is separate and untouched would prevent an implementer from conflating the two and editing the wrong file. Rationale: the two `.gitignore` locations are easy to confuse and the existing code already touches one of them.

## Summary

The PRD is complete enough to implement without major guessing: requirements trace cleanly to the BRIEF, ACs cover R1/R2/R4/R5/R6/R7, and the verified command surface and leak mechanisms are accurate. The notable gaps are a verification hole (R3's re-sync is asserted-by-transitivity but never exercised by an AC), an undefined-term risk around "niwa-authored file" that the generic test silently depends on, and a latent R5 contradiction tied to the deferred coverage-location design. None of these block the design phase, but the re-sync AC and the obsolete-warning question should be resolved before the DESIGN locks the recording mechanism.

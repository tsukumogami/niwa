# Architecture Review

## Verdict: PASS

The design is implementable from the Solution Architecture section alone, the six phases are correctly sequenced, and every PRD requirement R1-R26 maps onto a named component, decision, or phase deliverable. A small number of clarifications would help a fresh engineer, but none of them block a merge.

## Critical Issues

None blocking. The design's invariants (security defenses preserved, atomic snapshot, rank-2 single guard, leaf-package property for `internal/source`) are all stated and traceable to specific files. The probe/decide/extract/promote/disclose pipeline is internally consistent, and the data-flow diagrams (init, apply, `source inspect`) cover every entry point named in the PRD.

## Suggested Improvements

1. **Clarify `DisclosedNotices` scoping in the disclosure helper.** The PRD R14 calls for "once per workspace per artifact per command-type" and the design says the helper checks `notices.AlreadyDisclosed(id)`. The codebase actually has both per-instance state (`InstanceState.DisclosedNotices` in `internal/workspace/state.go:114`) and workspace-root state (loaded via `LoadState(workspaceRoot)` in `internal/workspace/apply.go:362`). Phase 5 should explicitly say which scope each rank-2 notice ID lives in. The natural fit is workspace-root state (so the notice fires once across all instances), but the design leaves this implicit and an implementer could pick the wrong scope and silently break AC-N3.
2. **Spell out the `disclosure.go` file location.** The Context section says "file location to be confirmed in Phase 4" but the Components section commits to `internal/workspace/disclosure.go` and Phase 5 ships it. Drop the "TBD" hedge in Context — it's already decided.
3. **Document `EnsureConfigSnapshotWithStatus` signature change explicitly.** Phase 4 says the signature grows a `rank int` return, but the Components section's pseudocode for `EnsureConfigSnapshotWithStatus()` doesn't show the new return tuple. A fresh implementer reading Components alone won't know the exact shape — show `(status SnapshotStatus, rank int, err error)` or equivalent.
4. **Add a `markers_probed` field to the `niwa source inspect` JSON success case.** The error cases include `markers_probed` (implicit in "probed both paths") but the success case only lists `markers_found_at_root`. For migration-skill diagnostics, showing what was *checked* (not just what was found) is useful and zero implementation cost.
5. **Pathological tarball case worth a one-line callout.** A tarball with 10M zero-byte entries fits well under the 500 MB decompression cap but still allocates substantial tar-header memory during pass 1. The design says "linear in tar entry count" in Consequences but doesn't acknowledge that header bookkeeping is a separate budget from `MaxDecompressedBytes`. Worth a one-sentence note or an explicit entry-count cap.
6. **`RankDecider` coupling concern is real but small.** The decider is shared across team-config (GitHub), team-config (non-GitHub), and overlay paths. Parameterization is via `MarkerSet` (clean) but the deprecation-notice message is hard-coded to "team config" / "overlay" via the notice-ID switch in the caller, not inside `RankDecider`. This is the right factoring; the design could just state this explicitly as a non-coupling argument (the decider returns rank, the caller picks the message). One sentence in Decision 1.
7. **`schema_version: 1` is robust enough but document the change policy.** The Consequences section says "future changes bump that field" — bumping on every breaking change is fine for an integer, but the design should commit to a policy: additive fields keep `schema_version: 1`, breaking changes (removed/renamed/retyped fields) bump it. Without this rule, tool authors don't know which changes are safe.

## Strengths

1. **Single rank-2 guard with BEGIN/END comments.** The follow-up release that hard-removes rank-2 becomes a one-function edit. This is exactly the right shape for deprecation hygiene and pays back the next time someone touches this code.
2. **Probe + extract + promote + disclose ordering is correct.** Notice emission *after* `SwapSnapshotAtomic` succeeds avoids the "stranded notice for a snapshot that never landed" failure mode. The design explicitly contrasts this with the R18 precedent and justifies why the precedent doesn't extend — that's clear thinking.
3. **`ProbeMarkers` factored out of `ProbeAndExtractSubpath` in Phase 6 instead of duplicating logic.** The CLI command and the production materializer share one probe primitive. The phase ordering (refactor lands when the second consumer arrives, not speculatively) is right.
4. **Overlay slug override is a leaf-package-internal change.** Decision 2 keeps `internal/source` free of imports from elsewhere in niwa, preserves the package's job ("encode current canonical identity"), and drops dead code. No deprecation cycle for a function with zero out-of-module callers is correct minimalism.
5. **Considered Options sections name and reject the right alternatives.** The single-pass stream-and-decide, the two-pass re-fetch, the gzipped-bytes buffer, and the embedded-callback designs are each rejected with concrete reasoning. The rejections hold up — none of them are attractive on a second look.
6. **PRD R1-R26 coverage is complete.** R1-R6 (discovery semantics) map to `RankDecider` and `MarkerSet`; R7-R9 (probe mechanism) map to `ProbeAndExtractSubpath` and `probeAndResolveCloneRoot`; R10-R12 (overlay) map to Decision 2 and overlay-path probe wiring; R13-R15 (deprecation) map to `disclosure.go` and the BEGIN/END guard; R16-R20 (skill) map to Decision 4; R21-R22 (compat) covered by the rank-2-still-resolves contract; R23 (rank-3 removal) is implicit — the design doesn't introduce rank-3, the diagnostic doesn't mention it; R24 (diagnostic clarity) is in the `RankDecider` error path; R25-R26 (reconciliation, docs) are Phase 6 deliverables.
7. **500 MB cap acceptance is correctly bounded.** The design acknowledges peak RAM grows to ~1 GB worst-case during `bytes.Buffer` geometric growth and accepts it — that's an honest trade-off statement, not hand-waving. The cap stays binding and observationally identical to today's behaviour.

## Summary

This design closes the R5 gap from the upstream config-sources PRD cleanly: a streaming probe rides on the existing fetch with no new round-trips, the rank-2 deprecation is gated behind one named constant with BEGIN/END delete markers, and the overlay derivation override is a leaf-package edit with zero downstream callers to migrate. The four decisions compose into a single coherent pipeline (probe → decide → extract → promote → disclose), and the six-phase plan is correctly sequenced — Phase 1's `RankDecider` has no live callers, Phase 6's refactor depends on Phase 2's `ProbeAndExtractSubpath` shape, and every intermediate phase ships with its own test surface using existing fixtures. The minor improvements suggested above (disclosure-scope clarification, `EnsureConfigSnapshotWithStatus` signature, schema-evolution policy) are documentation-quality fixes, not architectural concerns; the design is ready to implement.

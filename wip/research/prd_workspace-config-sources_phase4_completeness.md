# Completeness Review

## Verdict: FAIL

The PRD has solid coverage for most of its declared requirements, but it leaves several operationally important behaviors unspecified (notably the upgrade path for existing dot-niwa users whose source URL has not changed, the URL-change-detection mechanism that Story 2 narrates, and the workspace-name-mismatch failure mode that Phase 2 research surfaced). Several requirements also lack acceptance criteria.

## Issues Found

1. **Missing requirement for upgrade-without-URL-change conversion**: R23/R24 fire only when "the registry source URL has changed". The PRD never says what happens to an existing `org/dot-niwa` user whose URL is unchanged but whose `<workspace>/.niwa/` is still a working tree (has `.git/`). Goal "Preserve the existing standalone `dot-niwa` workflow" and R29 "No existing user MUST be required to take any action after upgrading" suggest the apply should continue working — but does niwa keep the working tree forever, lazily convert to snapshot, or trigger the same `--force` gate? AC-8 says it "resolves and applies successfully" but the on-disk shape is not pinned down. This is the most operationally important gap because every existing dot-niwa user lands here on first upgrade. Suggested fix: add a requirement explicitly stating the disposition (e.g., "first apply after upgrade lazily replaces the working tree with a snapshot, no `--force` required, with a one-time notice naming the conversion") or commit to "working tree continues to be supported indefinitely; conversion only happens on URL change."

2. **`niwa config set global` is invoked in Story 2 but never specified**: Story 2 reads "The developer runs `niwa config set global tsukumogami/vision`" as the trigger for the URL change. No requirement defines what `config set global` does, whether it exists today, whether it's part of v1, or what happens if the user instead edits `~/.config/niwa/config.toml` by hand (which Phase 2 maintainer-perspective AC-9 explicitly covers and the codebase-analyst's narrative #7 explicitly relies on). Suggested fix: either add a requirement covering the URL-change entry points (CLI command + manual file edit) or rephrase Story 2 to walk through the manual edit path.

3. **No requirement for workspace-name-mismatch validation during URL change**: Phase 2 maintainer-perspective AC-9.5 (and Capability-9 negative line) calls out a real failure mode: when the new source's `workspace.toml` declares a different `[workspace].name` than the registered name, niwa must refuse without `--rename` (or equivalent). The PRD's R23/R24 only handle URL change and disk shape; nothing prevents a developer from `--force`ing themselves into a workspace whose new source belongs to a completely different project. Suggested fix: add a requirement that on URL change, niwa validates the new source's declared workspace name matches the registry entry before atomically replacing the snapshot.

4. **No requirement for handling registry/mirror-field inconsistency**: Phase 2 maintainer AC-5.3 recommended: "When the registry contains an entry with parsed mirror fields that no longer match `source_url` ... niwa treats `source_url` as canonical, re-parses it, and overwrites the mirror fields on next save with a stderr warning." PRD has nothing on this. With both an opaque slug and parsed mirror fields persisted, hand-editing the registry file is a real possibility. Suggested fix: add a requirement covering canonical-source priority and the mirror-field reconciliation behavior.

5. **R3 strict parsing — only one of four cases has an AC**: R3 mandates rejection of (a) empty subpath after `:`, (b) malformed ordering of separators (e.g., `@ref` before `:subpath`), (c) embedded whitespace, (d) multiple `:` or `@` separators. AC-Subpath sourcing only covers case (a). Phase 2 UX Lead A specifically asked the PRD to enumerate rejected forms so they become testable. Suggested fix: add ACs for each of the other three rejection cases (empty subpath was the easy one; the others are where real users will mistype).

6. **R8's explicit `content_dir = "."` opt-in case has no AC**: R8 says `content_dir = "."` is valid (opt-in to "the whole brain repo is content"). The discovery AC tests only the failure case (omitted `content_dir` fails). No AC verifies that an explicit `"."` is accepted and that niwa then reads from repo root as content root. This is the explicit escape hatch the PRD chose to ship; it should be tested. Suggested fix: add an AC verifying the explicit `"."` case applies successfully.

7. **R34 (provenance marker human-readable) has no AC**: R34 commits to "readable with no specialized tooling — a future contributor inspecting `<workspace>/.niwa/` manually must be able to identify origin, ref, and fetched-at without running niwa." None of the ACs verify this property. The PRD also defers the file format and filename to design (Out of Scope), so it's the only gate ensuring the design choice is human-friendly. Suggested fix: add an AC verifying the provenance marker is parseable by `cat`/eyeball without niwa.

8. **R30 performance budgets have no AC**: R30 commits to "<5s first-time GitHub fetch on a typical (≤1 MB) subpath" and "<500 ms drift check". These are non-functional but they're the only quantitative commitments in the PRD. Without ACs (or at least an explicit "verified manually") they will silently regress. Suggested fix: either add benchmarks as ACs (`go test -bench`) or note that NFRs are verified informally and accept the looseness.

9. **No user story for non-developer roles surfaced in research**: The PRD has four developer-perspective stories. Phase 2 UX research and codebase analysis surface (a) a CI/automation operator who needs deterministic fail-on-stale (`--strict-refresh` is named in Decisions but not as a story), (b) a security reviewer running the public-repo plaintext-secrets guardrail (R26 covers it but no story narrates the inspection), (c) a developer running `niwa status` to inspect a workspace they didn't init themselves (Story 4 mentions niwa status output but in passing). At least the CI operator and security-reviewer stories matter because R19's "continue with cached snapshot when offline" and R26's "guardrail reads marker, not git remote" are both behaviors that hide silent regressions if the relevant user perspective is missing. Suggested fix: add a fifth story (CI operator) and possibly a sixth (security reviewer), or explicitly document why the PRD limits itself to developer scenarios.

10. **Repo-rename (301 from GitHub) is silently dropped from the PRD**: Phase 2 codebase-analyst Lead C narrative #6 and Lead D narrative #7 both call out the repo-rename case as a real edge case requiring a deliberate decision. PRD requirements and ACs say nothing about it — neither in scope nor in Known Limitations nor in Out of Scope. A repo rename is invisible to slug-based fetch unless the PRD commits to a behavior. Suggested fix: either add a requirement (follow-redirect-once-and-warn or hard-error) or explicitly mark "GitHub repo rename behavior" as Out of Scope so the design doc inherits the question.

11. **Authentication mechanism for private GitHub repos is unspecified**: AC-GitHub-tarball-fetch tests that 401/403 surfaces with a remediation hint, but no requirement states how niwa obtains the token (`GH_TOKEN` env var? `gh auth` integration? `~/.config/niwa/credentials.toml`?). The codebase already uses `GH_TOKEN` for `internal/github/client.go`; this should be ratified as the v1 token source. Suggested fix: add a requirement naming the token source and credential precedence (env var, then existing helper, then anonymous).

## Suggested Improvements

1. **Tighten R13 ACs**: AC for R13 currently verifies only the no-`.git/`, atomic-refresh, marker-present properties for personal and workspace overlay clones. It doesn't verify that personal/workspace overlays also use the GitHub-fast-path / fallback split (R14/R15 apply to "the source"; less clear they apply to overlays). Add an AC explicitly covering "the personal overlay clone uses the GitHub tarball path when its source is github.com."

2. **Cross-reference R8's "rank 3" wording**: R8 reads "When discovery resolves to repo root via the rank-3 `niwa.toml` convention". R5 also calls rank-3 `niwa.toml` and rank-2 `workspace.toml`. AC-Convention-based discovery says "When the source repo contains only a root `niwa.toml`, discovery resolves `source_subpath = ""`" — consistent. But "rank 3" in the requirements is a layered concept that an implementer parsing the requirements alone won't have a glossary for. A small "Discovery rank glossary" subsection would help.

3. **Make Story 2's `niwa apply --force` UX consistent with R23**: R23 wording says the error must "suggest an inspection command (e.g., `cd .niwa && git status`)". Story 2 prose says "instructing the developer to inspect `<workspace>/.niwa/` for pending edits". The story matches the spirit but not the literal command. Consider quoting the exact command in Story 2 so the user-facing spec and the requirement match verbatim.

4. **Out of Scope: add explicit entry for `--strict-refresh`**: The Decisions section mentions `--strict-refresh` as a follow-up candidate for CI users, but the Out of Scope section doesn't list it. Adding it makes the deferred surface explicit and prevents scope creep into v1 review cycles.

5. **R20 lazy upgrade — clarify trigger**: R20 says mirror fields are "lazily upgraded by populating the mirror fields on the next registry write." But what counts as a "registry write"? `niwa create` triggers one; pure-read commands like `niwa status` don't. A reader might assume `niwa apply` always triggers a write and so migration "just happens". Worth a one-sentence clarification.

6. **Known Limitations: snapshot-edit warning**: The "manual edits silently discarded on refresh" limitation could be elevated to an active behavior (e.g., niwa MAY emit a warning if it detects newer mtimes on snapshot files than the marker's `fetched_at`). Phase 2 codebase-analyst Lead C touched on this in the "snapshot corruption" discussion. Currently it's a documented limitation only.

## Summary

The PRD is structurally sound and most requirements have matching acceptance criteria, but it leaves three operationally important gaps unaddressed: the upgrade path for existing dot-niwa users whose URL hasn't changed (Issue 1), the URL-change-detection mechanism that Story 2 narrates without specifying (Issue 2), and the workspace-name-mismatch validation during URL change that Phase 2 research surfaced as a real failure mode (Issue 3). Severity is medium-to-high — implementers can guess at most of these, but the guesses risk creating regressions for the very users the PRD's "Preserve the existing standalone `dot-niwa` workflow" goal protects. Recommend revising to add the three missing requirements, the four missing ACs (Issues 5-8), and at minimum the CI-operator user story; the remaining items can be folded in as polish.

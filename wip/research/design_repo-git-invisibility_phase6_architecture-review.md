# Architecture Review

**Verdict:** PASS

The mechanism (write a delimited niwa block into the common-dir `info/exclude`, resolved via `git rev-parse --git-common-dir`) is correct and verified against real git for both primary checkouts and linked worktrees; the only defects are inaccurate prose about *where* `CheckGitignore`/`SaveState` run and an imprecise path/fixture name, none of which block implementation.

## Findings

1. **Core mechanism is sound (verified empirically).** I reproduced the design's claim end-to-end: writing a `# >>> niwa managed >>>` / `# <<< niwa managed <<<` block with `.niwa/` into `<repo>/.git/info/exclude` makes `.niwa/` invisible to `git status` in a linked worktree, because `git -C <wt> rev-parse --git-common-dir` resolves to the shared `<repo>/.git`. One write per repo genuinely covers the primary checkout and every worktree. Decisions 1-3 (exclude surface, fixed `*.local*`+`.niwa/` block, delimited-block idempotency) are correct and well-justified.

2. **`--git-common-dir` relative/absolute handling is correct *as used*, but the design's framing is slightly under-specified (minor correctness note).** For a primary checkout git returns the **relative** string `.git`; for a linked worktree it returns an **absolute** path. The design says "resolve the result relative to `tree` when it is not absolute." This works only because git is invoked with `-C <tree>` and the relative output is relative to that `-C` directory — and because both call sites pass the repo/worktree **top-level** as `tree`. I confirmed that from a nested subdir git would instead return `../.git`; the design is safe purely because callers never pass a subdir. Recommend the implementation use `filepath.Join(tree, commonDir)` + `filepath.Clean` (which correctly handles a `..`-bearing result too) and a one-line comment stating the invariant "tree must be the working-tree top-level." No functional gap for the two real call sites.

3. **The `CheckGitignore` removal description is factually wrong and will mislead the implementer (the main defect).** The design states the obsolete warning runs "the same place `SaveState` and the existing `CheckGitignore` warning run" and locates it in `content.go`. In reality `CheckGitignore` is invoked from **two** in-pipeline spots, neither adjacent to `SaveState` (which lives in `Apply`/`Create`, not `runPipeline`): `InstallRepoContent` (content.go:140 and :177) and `SettingsMaterializer.Materialize` (materialize.go:523). An implementer following the design literally would remove one site and miss the other, leaving a contradictory warning still firing (violating R6 intent). The removal must cover both call sites plus their tests (`content_test.go`, `materialize_test.go`).

4. **Apply call-site location is implementable but mislocated in the design's prose.** `EnsureRepoExclude(repoDir)` "once per managed repo after materialization" cannot go "where SaveState runs" — `SaveState` is one workspace-level call, not per-repo. The correct home is the existing per-repo loop in `runPipeline` Step 6.5 (apply.go ~1344-1409), where `repoDir := filepath.Join(instanceRoot, cr.Group, cr.Repo.Name)` is already computed and materializers have just written the `.local` files. Returning an error there propagates up through `runPipeline` -> `Apply`/`Create` and aborts the apply (fail-closed, R9) — the design's intent holds, but the named anchor (`SaveState`/`CheckGitignore` colocation) does not exist. Implementer should target the Step 6.5 loop (or a dedicated post-materialization loop over `classified`).

5. **Worktree call-site error routing is correct.** In `CreateSession` (handlers_session.go:314-317), after `scaffoldWorktreeNiwa(wtPath, params.Repo)` succeeds, calling `EnsureRepoExclude(wtPath)` and on error doing `cleanupWorktree(); return "", "", branch, fmt.Errorf(...)` exactly mirrors the existing `scaffoldErr` arm. The cleanup removes the worktree via the injected `GitInvoker`, so the design's "joins the existing cleanup-on-failure path" is accurate. The data-flow diagram's `git -C <wt> rev-parse --git-common-dir => <repo>/.git` is verified.

6. **Test fixture names/paths are slightly off (minor).** The design references a `localGitServer` factory and `git-invisibility.feature` at `test/functional/`. Actual: the constructor is `newLocalGitServer` returning `*localGitServer` (test/functional/localrepo_test.go:19), `runGitInDir` exists (session_steps_test.go:782), but feature files live under `test/functional/features/` — so the new feature belongs at `test/functional/features/git-invisibility.feature`. The negative/idempotency/user-content scenarios are well-specified and the fixtures support them.

7. **Phase sequencing is correct.** Implement `exclude.go` + pure-renderer unit tests first (no git needed), then wire apply (+ remove `CheckGitignore`), then worktree, then functional feature, then verify. `renderNiwaBlock` as a pure `[]byte -> []byte` function is genuinely unit-testable without a repo and is the right seam. No reordering needed.

8. **No simpler alternative was overlooked.** The four rejected options (committed `.gitignore`, `skip-worktree`/`assume-unchanged`, global `core.excludesFile`, derive-from-file-set) are each correctly ruled out against R3/R4. `info/exclude` is the unique surface that is repo-scoped, never committed, and worktree-shared. The fixed broad block + behavioral test (rather than per-file derivation) is the right call given the prior `ManagedFiles`-registry blind spot.

## Recommendations

1. Correct the prose in "Decision Outcome" and "Call sites": the apply call belongs in the `runPipeline` Step 6.5 per-repo loop (where `repoDir` exists), not "where `SaveState`/`CheckGitignore` run."
2. Fix the `CheckGitignore` removal scope to name **both** call sites (`content.go` InstallRepoContent x2, `materialize.go` SettingsMaterializer) and their tests; the current single-location description risks an incomplete removal.
3. In `EnsureRepoExclude`, use `filepath.Join(tree, commonDir)` then `filepath.Clean`, document the "tree = working-tree top-level" invariant, and create `<commonDir>/info` with `MkdirAll` before writing.
4. Correct the test path to `test/functional/features/git-invisibility.feature` and the fixture name to `newLocalGitServer`.

## Summary

The architecture is implementable as written: the common-dir `info/exclude` write is the correct, empirically-verified surface for both primary checkouts and worktrees, the pure-renderer/helper split is clean, and the worktree error routing matches the existing cleanup pattern. The blocking-adjacent issues are documentation inaccuracies — chiefly that `CheckGitignore` runs at two in-pipeline sites (not "where SaveState runs"), so the removal scope and the apply call-site anchor are misdescribed. These are prose fixes, not design gaps, so the verdict is PASS with the recommendation that the implementer treat findings 3 and 4 as authoritative over the design's stated locations.

# Completeness Review

**Verdict:** PASS

The PRD covers every corner case the BRIEF named and the review checklist demanded, with each launch location, teardown path, capture failure mode, and partial-failure step pinned to an explicit SHALL requirement and a matching acceptance criterion.

## Gaps Found
1. Self-dispatch / nesting depth bound is under-specified: The BRIEF's Scope Boundary "In" list explicitly names "a dispatched worker itself invoking the command (nesting)" as a corner case the command must survive. The PRD addresses nesting only for the *human* dispatcher inside an instance (R6: resolve enclosing root, create sibling, never nest) and for the re-entrancy hook no-op (R39). It does not state what happens when a *dispatched background worker* runs the dispatch command from inside its own instance — R6's sibling rule implies it would create another sibling under the same root, but the PRD never confirms this is intended, nor whether recursive fan-out is bounded or guarded. This is a named BRIEF corner case that lands in an implied-but-unstated state. Suggested requirement: an explicit SHALL that a dispatch invoked from within a dispatched worker's instance resolves to the same workspace root and creates a sibling (same as R6), with no special recursion guard required, or a note that recursion depth is the developer's responsibility.

## Suggested Improvements
1. Make the empty-vs-whitespace prompt boundary explicit in R43: R43 rejects "an empty prompt" but does not say whether a whitespace-only prompt counts as empty. A one-clause clarification would remove an implementation ambiguity that could otherwise leak into the DESIGN.
2. State the label-collision / label-validation behavior for R2/R25: the optional human-friendly label is recorded on the mapping but the PRD does not say whether labels must be unique, may collide across concurrent dispatches, or are sanitized. Since R36 guarantees distinct instances by token (not label), a one-line note that the label is purely informational and non-unique would close the question the DESIGN will otherwise ask.
3. Cross-reference R30's "sufficiently-stale" liveness backstop to the Known Limitations TTL note: the reboot/crash reclamation in R30 leans on the stale-job-state backstop, which the Known Limitations section flags as a possible over-eager reclaim of idle sessions. A pointer between the two would make the trade-off self-documenting for a future reader.

## Summary
The PRD is exhaustive on the dimensions the review demanded: all five launch locations (R5-R9), all five teardown paths (R27-R31), every capture failure mode (R18-R22), per-step partial-failure atomicity with a no-unreclaimable-orphan guarantee (R32-R35), concurrency including the in-flight-not-swept protection (R36-R38), hook coexistence (R39-R41), argv-only prompt limits (R43), and the additional cases — ephemeral-mode independence (R4), claude preflight (R16), env materialization (R10), and provenance marker (R24). All three BRIEF Open Questions and every Scope-Boundary "Out" item are closed in Decisions/Trade-offs and Out of Scope. The single genuine gap is that the BRIEF's named "nesting — a dispatched worker invoking the command" corner case is addressed for the human-inside-an-instance case but never explicitly resolved for a worker self-dispatching; it is implied by R6 but not pinned, which is a requirement-surface gap worth one sentence rather than a blocker.

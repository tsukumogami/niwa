# Completeness Review

## Verdict: PASS

Every requirement carries at least one acceptance criterion and every BRIEF journey and Scope In-item traces to a requirement, but four real gaps (secret revocation on re-mint, team-phase re-run verification, initial-session/overlay-creation preconditions, and partial-failure cleanup) should be closed before the DESIGN inherits them.

## Issues Found

1. **Secret revocation on re-mint is a dangling half-requirement.** R8 step 2 mandates that the minted secret's id "MUST be captured so a later run can revoke it," but no requirement operationalizes the revocation itself: nothing says whether, when, or under what conditions a re-run (R15/US-6 re-mint-and-re-store, or a topology change) revokes the prior secret. AC-21 verifies re-mint/re-store but not that the superseded secret is revoked or that the wizard even attempts it. The capture is stored state with no consumer in the spec. Fix: either add a requirement + AC that a re-run revokes (or best-effort revokes, with a stated failure behavior) the prior minted secret it recorded, or move revocation to Out of Scope explicitly and drop the "so a later run can revoke it" clause. As written an implementer must guess whether accumulating dead secrets across re-runs is acceptable.

2. **Team-phase re-run verification is underspecified.** US-5 and BRIEF journey 4 both state a team admin re-runs `niwa onboard` to "confirm the team-phase setup before teammates depend on it." But R11 — the only wizard-end verification — is defined entirely in terms of the individual-phase credential-sync read topology (three vault-registry sources, self-exclusion, contract validator), which does not exist for a team admin who has produced an identity/auth/grant/folders and no personal-overlay credential. R15 says a completed setup "MUST go straight to the wizard-end verification (R11)," but R11 has no meaning for the team setup. Fix: add a requirement (and AC) defining what verification a completed *team* setup runs on re-run — presumably re-confirming the identity exists and exposes a `client_id`, the grant is present, and the folders exist, mirroring R6's per-step landing checks — so the admin half of US-5 is actually buildable.

3. **Initial-session and personal-overlay-creation preconditions are unowned.** R4/R5 assume "the operator's initial session is established" and R8 mints "against the org that hosts the workspace vault," but no requirement makes the wizard ensure the operator is authenticated at start or specify what happens if they are not logged in when `niwa onboard` runs. Separately, US-2 and BRIEF journey 2 say the wizard "sets up my personal overlay," yet the requirements only cover *writing into* the personal-overlay repo (R12); none owns creating or bootstrapping the overlay if it does not yet exist. Fix: state whether an authenticated session and an existing personal-overlay repo are preconditions the wizard checks-and-fails-fast on, or steps the wizard performs. This is the difference between an implementer building overlay creation and assuming it exists.

4. **Partial-failure cleanup (mint succeeds, store fails) has no stated behavior.** R16 assigns a distinct exit code to storage-write failure, and R15 covers resume, but nothing specifies what happens to a freshly minted-and-verified secret when the subsequent store fails: is it revoked, left for the next run to reuse, or orphaned? This is the concrete instance of gap #1 and intersects the resume story. Fix: specify the on-store-failure disposition of the just-minted secret (tie it to the revocation requirement in #1).

## Suggested Improvements

1. **Enumerate the non-interactive input surface referenced by R18.** R18 fails fast when "the inputs it needs were not supplied non-interactively," implying flags beyond the R2 setup-override exist, but the PRD never lists them. A one-line pointer to "the setup-selection and topology-selection overrides (R2, R3) are the non-interactive inputs" would remove ambiguity without pulling DESIGN mechanics forward.

2. **Add a negative AC for a failed guided-step verification.** AC-9 covers the positive case (wizard verifies the step landed before continuing); a companion AC asserting the wizard does *not* continue when the operator claims a guided dashboard step is done but the verification (e.g. identity absent / no `client_id`) fails would harden R6's guarantee, which is the crux of the "honest, unglamorous" degradation.

3. **Clarify that AC-26's five exit codes are exhaustive vs. representative.** R16 lists five outcome classes; if a guided-step-abandoned outcome differs from generic "operator decline/abort," name it, otherwise state that abandonment at a guided step folds into decline/abort so the mapping is unambiguous.

## Summary

The PRD is strongly built: all 19 functional/non-functional requirements have acceptance-criterion coverage, all six BRIEF journeys and every Scope In-item map to requirements, and Out of Scope is unusually clear (five explicit exclusions plus the hard admin-custody line). The material gaps are a dangling revocation requirement (R8 captures state nothing consumes), an individual-phase-only wizard-end verification that leaves the team-admin re-run journey (US-5) without a defined check, and two unowned preconditions (initial session, personal-overlay creation) that the journeys assume. None blocks the core individual-setup happy path, so this passes, but closing the four issues will prevent the DESIGN and implementer from guessing at re-run and failure behavior.

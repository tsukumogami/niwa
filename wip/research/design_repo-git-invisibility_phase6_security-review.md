# Security Review

**Verdict:** PASS

The design's write target derives entirely from git's own path resolution over a `.git` directory niwa already owns and writes to, with no untrusted interpolation; the residual vectors (symlinked exclude file, repo-config-influenced rev-parse) are real but sit inside a trust boundary niwa already crosses, so none rises to a blocker.

## Findings

1. **Path traversal via git-dir resolution — adequately handled, with one nuance the design states correctly.** `EnsureRepoExclude` resolves the exclude file via `git -C <tree> rev-parse --git-common-dir` and appends the fixed literal `info/exclude`. I verified empirically that from a primary checkout `--git-common-dir` returns the *relative* `.git`, while from a linked worktree it returns an *absolute* path (`/tmp/rvtest/.git`). The design's step 1 ("resolve the result relative to `tree` when it is not absolute") is therefore load-bearing and correct — a naive `filepath.Join` without the relative-resolution branch would still work for the relative case but the design calls it out explicitly. No attacker-controlled string enters the path; the only inputs are git's own output and a fixed literal. Not a vector.

2. **Repo git config can redirect rev-parse output (residual, in-boundary).** `git rev-parse --git-common-dir` honors `GIT_COMMON_DIR`/`GIT_DIR` env vars and a repo's own config, so a maliciously crafted repo could in principle point the common-dir elsewhere and cause niwa to write `info/exclude` into an attacker-chosen git dir. This is NOT called out in the Security Considerations section by name. However: (a) niwa only operates on repositories it clones/manages, (b) the written content is two fixed ignore patterns (no payload), and (c) the env is niwa's own process env, not repo-supplied. The worst case is writing a harmless two-line ignore block into a redirected `.git/info/exclude`. Acceptable residual; worth a one-line note but not a blocker.

3. **Symlinked `.git/info/exclude` — real gap in the stated mitigation, low severity.** `os.WriteFile` follows symlinks. If `.git/info/exclude` (or `info/`) is a pre-existing symlink pointing outside the git dir, the niwa write follows it and writes through. The design's "Symlink consideration" bullet dismisses this as "no new traversal surface beyond what niwa already has over `.git`," which is fair reasoning — placing a hostile symlink inside `.git/info/` already requires write access to the repo's git dir, which is the same access niwa exercises for state and scaffolding. The justification is applicable, not hand-waved. A defense-in-depth `Lstat`-and-reject-symlink check would harden it but is not required to PASS.

4. **TOCTOU between read and write — present but benign.** The helper reads existing content, splices the managed block, then writes with `os.WriteFile` (non-atomic in the sense of read-modify-write; a concurrent writer between read and write loses). The design names an "atomic" write in the Solution Architecture but `os.WriteFile` is not atomic (no temp-file-rename described). Concurrent `niwa apply` + `session create` on the same repo both target the same common-dir file and could race, with last-writer-wins clobbering the other's managed block. Both writers produce the *same* managed block, so the converged content is correct regardless of ordering; the only loss would be interleaved user content added between the two operations, which is not a realistic concurrent-edit scenario. Recommend a temp-file + rename to honor the "atomically" claim, but the failure mode is benign given identical block content.

5. **Fail-closed posture is sound.** An unwritable exclude file aborts the apply / rolls back the worktree (joins the existing cleanup path). This correctly prevents the "silent leak" failure mode, which is the actual security-relevant property of the feature (preventing secret-bearing `*.local*` files from showing as untracked and being accidentally committed). The mitigation matches the risk.

6. **"No secret handling / 0o644 permissions" justification is correct.** The block contains only glob patterns, no secrets, so world-readable `0o644` on the exclude file is appropriate and consistent with git's own default perms for that file. The "not applicable" framing for secret handling is genuinely not applicable. Note the *purpose* of the feature is secret hygiene (hiding `*.local*` secret files from git status), but the exclude file itself holds no secrets — the design draws this distinction correctly.

## Recommendations

1. Add one sentence to Security Considerations acknowledging that `rev-parse --git-common-dir` resolution trusts the managed repo's git config/env, bounded by niwa already owning these repos and the payload being two fixed patterns (Finding 2).
2. Either implement the write as temp-file + atomic rename to match the "Write atomically" claim, or soften that wording to "write in place" (Finding 4). Identical block content makes the race benign, so this is correctness-hygiene, not a security blocker.
3. Optional defense-in-depth: `Lstat` the exclude path and refuse to follow if it is a symlink, converting a surprising-but-in-boundary write into an explicit fail-closed error (Finding 3).

## Summary

The security posture is sound: the write target is derived from git's own resolution of a `.git` directory niwa already owns and writes to, the content is two fixed literals with no untrusted interpolation, and the fail-closed design correctly prevents the only security-material outcome (silently leaking secret-bearing files into git status). The unaddressed vectors — config-influenced rev-parse, symlinked exclude file, and a benign read-modify-write race — all sit inside the existing niwa-owns-`.git` trust boundary and carry no exploitable payload, so they are residual hardening items rather than blockers. Recommend documenting the config-trust assumption and reconciling the "atomic write" wording before implementation.

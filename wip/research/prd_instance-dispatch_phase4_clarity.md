# Clarity Review

**Verdict:** PASS

The PRD has all required sections in canonical order, frontmatter that matches the body, numbered and independently-understandable requirements, and it holds WHAT/WHY altitude while staying internally consistent and free of style violations.

## Issues Found
1. R14 and R18 spell `claude --bg` and the literal `backgrounded · <short-id>` output format as part of the requirement text: This is borderline HOW. R14's "launch a Claude Code background session" is acceptable WHAT, but pinning the exact invocation flag (`--bg`) and the human-formatted output string elevates an implementation detail into a SHALL. The format string is correctly hedged ("which today is human-formatted ... with no machine-readable mode") and the Known Limitations section already flags scrape fragility, so this reads as motivating context rather than a mandate — it does not sink the verdict, but the DESIGN, not the PRD, owns the exact launch surface. Suggest softening to "launch a Claude Code background session (e.g. the `claude` background-launch surface)" and moving the literal output example fully into a parenthetical rather than the normative clause.
2. R19/R21 reference "Claude Code job state" and "the job state's recorded working directory" as the capture mechanism: This sits right at the altitude line. "Capture the id from job state" is explicitly named as acceptable WHAT in the review brief, and the PRD never names a function or package, so it passes — but R21's "for example by correlating the instance directory ... against the job state's recorded working directory" leans toward prescribing the algorithm. It is correctly framed as an example ("for example"), keeping it advisory rather than mandatory, so it stays on the WHAT side.

## Suggested Improvements
1. Make the optional-section ordering explicit relative to Out of Scope: Decisions and Trade-offs, Known Limitations, and References surround the required Out of Scope section. The seven required sections are in correct relative order, but a reader skimming for the canonical tail (Out of Scope last) finds References after it. Consider moving References to the very end and grouping the optional sections before Out of Scope for cleaner skimming. Rationale: aids the "Out of Scope closes the requirement surface" reading.
2. R16 ("verify the `claude` executable is available") and R44 ("add no new system dependency beyond what `niwa create` and a local `claude` already require") together imply `claude` is a hard prerequisite; state that dependency once, up front in the command-surface group, so the precondition is not inferred across two distant requirements. Rationale: improves independent-understandability of R16.
3. R11's rationale ("no concurrency-safe reservation in the numbered-instance naming scan") narrates an existing implementation weakness inside a requirement: The WHAT ("name each instance from a freshly-generated unique token ... so two simultaneous dispatches cannot collide") is clean; the preamble explaining why the existing scan is unsafe is design reasoning that already lives, correctly, in Decisions and Trade-offs. Trimming the preamble from R11 keeps the requirement at altitude. Rationale: avoids duplicating the trade-off rationale into the normative text.

## Summary
The PRD is well-structured and internally consistent: required sections are present and ordered, the frontmatter status matches the bare "Draft" body line (FC03), requirements are numbered R1-R45 and individually testable, and the additive/no-modify-the-hook promise in the Goals, R1, R39-R41, and Out of Scope is held without contradiction. Style is clean — no banned words, no emojis, no AI attribution, and the only external refs are public (tsukumogami/niwa#171, #172; anthropics/claude-code#60975, #31940). The one altitude wobble is R14/R18 spelling the exact `claude --bg` flag and the literal launch-output format as normative, but both are hedged and the corner-case surface is otherwise correctly pitched at WHAT, so the verdict is PASS.

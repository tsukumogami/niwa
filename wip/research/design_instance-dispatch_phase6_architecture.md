# Architecture Review

**Verdict:** PASS

The design satisfies the PRD's hard requirements with a sound create->launch->capture->map->unmark sequence and a correctly-gated marker+TTL backstop, but the prose describing the reaper extension understates a real structural detail that the plan must get right.

## Issues Found

1. **The backstop branch cannot live "inside" `selectReapTargets`'s existing record loop as the design implies (Solution Architecture, lines 226-229).** `EnumerateInstanceRecords` (`internal/workspace/state.go:406-426`) derives `InstanceRecord.Ephemeral` *only* from the mapping store via `ephemeralInstancePaths` (`state.go:437-467`), which scans `.niwa/sessions/*.json`. An unmapped instance is therefore always `Ephemeral:false`, and `selectReapTargets` drops it at the very first guard (`reap.go:112`, `if !rec.Ephemeral { continue }`) before any new branch could run. The design's phrase "an on-disk ephemeral-by-marker instance" papers over the fact that there is *no* on-disk ephemeral signal independent of the mapping today (D4 report lines 17-22 says exactly this). Why it matters: a naive reading of "extend `selectReapTargets` with one branch" produces a backstop that never fires, because the orphan it targets is invisible to the enumerator it iterates. Fix: the backstop must be a *separate* enumeration — scan instance directories for the dispatch pending-marker directly (or via a new marker-aware enumerator), then apply the mapping-absent + age>TTL gate to that scan. This is still additive and does not change the existing mapped+liveness path, so the decision is sound; the plan/prose just needs to name the separate scan rather than imply an in-loop branch.

2. **Marker-age source is unspecified, and `state.json`-style mtime/embedded-timestamp choice has a correctness consequence for R38 (Solution Architecture line 214; D4 lines 178-180).** The TTL gate that preserves R38 depends entirely on a trustworthy "marker age." The design says the marker records a creation timestamp (D4 line 172) but the architecture section describes only "a small file ... `.niwa/dispatch-pending`" without fixing whether age comes from file mtime or an embedded stamp. Why it matters: filesystem mtime can be altered by tooling (rsync, archive restore, clone), and a wrongly-young mtime would let the backstop spare a true orphan forever while a wrongly-old one could reap an in-flight instance (R38 violation). Fix: embed an explicit RFC3339 creation timestamp inside the marker file and gate on that, not on mtime; the plan should state this.

## Suggested Improvements

1. **Make the marker's mapping-absent gate read the same store the reaper already joins on, not a fresh stat.** The backstop's safety rests on "mapping absent." `selectReapTargets` already builds `byPath` from `ListSessionMappings` (`reap.go:96-105`); the backstop scan should consult that same map so the two reclamation paths share one definition of "mapped." Rationale: prevents a TOCTOU split where the liveness path and the backstop path disagree about whether an instance is mapped, and keeps the never-reap-a-mapped-instance invariant in one place.

2. **State explicitly that the backstop scan must exclude hook-created instances by the marker, not by origin.** Hook instances have a mapping and no dispatch marker, so they are already excluded by both gates — but the design's own D6 says `Origin` is informational and the reaper ignores it (lines 157-159). The plan should assert the backstop keys on *marker presence*, never on `Origin == dispatch`, so the provenance field stays purely informational and R41 holds. Rationale: prevents a future maintainer from "optimizing" the backstop to filter on `Origin`, which would couple reclamation eligibility to provenance and break R41.

3. **Name the marker-cleanup-on-success ordering as load-bearing in the plan's step list.** The rollback ordering (mapping written BEFORE marker removal, line 241 step 6) is correct and is what guarantees a mapped instance is never the backstop's target — but it is stated only as a data-flow step. Rationale: this ordering is the single invariant that prevents a mapped-but-still-marked instance from being double-counted; a plan that reorders steps 5 and 6 would reintroduce the orphan window, so it deserves an explicit "do not reorder" note.

## Summary

The design fits niwa's architecture cleanly: it reuses `realProvisionInstance`/`applier.Create`, `destroyInstanceFunc`, `WriteSessionMapping`, `ClassifyCwd`, the `jobsDir` injection seam, and the sessionattach exec pattern, all of which I verified exist and are used as described; the additive `Origin` (on `SessionMapping`) and `Cwd` (on `jobState`) fields are genuinely backward-compatible (zero-value/omitempty decode, no existing reader breaks). The create->launch->capture->map->unmark sequence and its rollback ordering are correct — the mapping is durable before the marker clears, so a mapped instance is always reclaimable and the TTL-gated backstop only fires on mapping-absent young-excluded orphans, preserving R38 and not touching hook-created (mapped, unmarked) instances. The one substantive correction is structural-descriptive: the backstop is not an in-loop branch of `selectReapTargets` (the enumerator's `Ephemeral` flag is mapping-derived, so an unmapped orphan never reaches the loop body) but a separate marker scan with an embedded-timestamp TTL gate — a fix the downstream plan must encode, not a flaw in the decision itself.

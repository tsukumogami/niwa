# Clarity Review

## Verdict: PASS

The PRD is sufficiently specific that two developers building from it should converge on the same behaviour; remaining ambiguities are localised and resolvable without re-architecting.

## Ambiguities Found

1. **R1 / AC-D1 ("source root" scope)**: R1 talks about probing "the source for marker files" while R3 / AC-D5 narrow this to "the source root". The PRD never explicitly says discovery probes only the top-level directory of the source tarball (e.g., that a nested `subdir/.niwa/workspace.toml` is invisible to discovery). -> A developer could implement a recursive scan for `workspace.toml` anywhere in the source. -> Add a sentence in R1: "Discovery inspects only entries directly under the source-root directory; nested occurrences of these filenames at deeper paths MUST be ignored."

2. **R6 (non-GitHub probe semantics)**: "single shallow clone (non-GitHub path)" -> What does "scan the temp-clone directory" mean concretely for the marker-detection contract? Is it `git ls-tree HEAD --name-only` filtered to the two paths, or a filesystem walk? Does the shallow clone happen before or after probe (since clone of a huge repo cannot be aborted by discovery failure)? -> A developer could implement either a tree-listing probe or a full-checkout-then-walk; the cost profile differs significantly. -> Specify: "For the non-GitHub path, niwa MUST perform a single shallow clone (`git clone --depth=1`) and inspect the resulting working tree's top level for marker files; the clone is reused as the staging source if discovery resolves successfully."

3. **R8 (`--to` omitted, non-interactive mode without `--yes`)**: The PRD specifies behaviour for interactive (prompt) and non-interactive-with-`--yes` (apply directly), but leaves non-interactive-without-`--yes` undefined. -> A developer could exit zero with a printed suggestion, exit non-zero, or apply silently. -> Clarify: "In non-interactive mode without `--yes`, `niwa migrate-source` MUST print the suggested slug and exit non-zero with a message instructing the user to re-run with `--yes` or `--to <slug>`."

4. **R8 "single unambiguous migration target"**: The phrase is undefined. What constitutes "unambiguous" when probing the *current* source — is it "exactly one marker found" (which is always true for any source that currently resolves), or something stronger? -> Two readers will disagree on when the no-`--to` form succeeds. -> Define: "Discovery against the current source produces a single resolved subpath via R1's precedence; the inferred new slug is `<owner/repo>:<resolved-subpath>` or `<owner/repo>` if the resolved subpath is empty. If discovery fails (R3/R4), `niwa migrate-source` without `--to` MUST exit non-zero with the discovery diagnostic."

5. **AC-D5 ("removed or unchanged if pre-Mkdir")**: "The on-disk `<workspace>/` is removed (or unchanged if the init was aborted pre-Mkdir)" -> This dual outcome makes the AC non-binary: a reviewer cannot tell which state to assert without knowing implementation order. -> Make it binary by asserting the contract directly: "After AC-D5's failure, `<workspace>/` either does not exist or is byte-identical to its pre-init state. The test MUST assert that no files written by the init are present."

6. **AC-X1 / AC-X2 "three distinct lines" / "four distinct lines"**: The acceptance criteria require N substrings on N distinct lines, but R3 / R4 / R14 examples in the Stories show formats where the slug appears on the same line as the `error:` prefix (e.g., `error: ambiguous niwa config in acme/vision`). Counting that as a "distinct line for the slug" is plausible but not asserted. -> Reviewers may disagree about whether `error: ambiguous niwa config in acme/vision` satisfies "the source slug on its own line". -> Soften to: "MUST contain three substring matches that may appear on the same or different lines: the source slug, `.niwa/workspace.toml`, `workspace.toml`" — or commit to the example format from Story 6 / 7 verbatim as the golden.

7. **R14 "every accepted marker path that was probed"**: For the R3 ambiguity case, only the markers actually present are conflicting — the PRD does not say whether the error names *all probed* markers or only *the markers found*. -> AC-D5 says "both file paths" (the two that were found, in the ambiguity case); AC-X1 says "every accepted marker path that was probed". These are the same in the two-rank case but the wording diverges. -> Tighten R14: "For ambiguity errors, the message names every marker found; for no-marker errors, the message names every accepted marker path."

8. **R5 / AC-D7 ("byte-identical to pre-init state")**: For `niwa init`, the workspace directory typically doesn't exist before the command runs. "Byte-identical to its pre-init state" is well-defined for re-init scenarios but underspecified for first-time init. -> A reviewer cannot verify byte-identity of a directory that didn't exist. -> Reword: "MUST leave the on-disk `<workspace>/.niwa/` byte-identical to its pre-init state, or absent entirely if it did not exist before."

9. **R10 wording "refuse without `--force` per upstream R26"**: This re-states upstream behaviour but is phrased as a fresh requirement. -> A developer might read R10 as adding new gate logic when in fact it asserts the upstream gate still fires. -> Reword: "Apply-time behaviour after a `migrate-source`-driven source-URL change MUST be identical to the upstream PRD's R26/R27 (no new gates added by this PRD)."

10. **AC-M3 ordering / preconditions**: "Given a workspace registered with `source_url = "acme/dot-niwa"` AND a source that has been rebased to host its config at `acme/vision/.niwa/`" — the AC describes a precondition involving two distinct sources (the legacy `acme/dot-niwa` and the new `acme/vision`) but does not name the second `tarballFakeServer` setup. -> A test author cannot derive the fixture setup unambiguously. -> Specify: "Two `tarballFakeServer` fixtures are required: one serving `acme/dot-niwa` (the legacy standalone shape) and one serving `acme/vision` (with `.niwa/workspace.toml`)."

11. **R13 / AC-R1 vs. Known Limitations**: R13 mandates removing rank-3, and the Known Limitations section calls the removal a "small breaking change". For users with an existing registry pointing at a `niwa.toml`-rank-3 source, the PRD does not say whether the next `niwa apply` errors out (the implication of AC-R1) or whether an upgrade-time pre-flight catches it. -> A reader could assume either a silent failure on apply or a guarded upgrade path. -> Add a sentence to R13: "Existing registries whose source resolves only via the removed rank-3 marker MUST surface the R4 'no niwa config found' diagnostic on the next `niwa apply`; no upgrade-time prompt or pre-flight is added."

12. **Open Questions Q1-Q4 in a Draft PRD**: The PRD's status is "Draft" and Open Questions are explicitly required to be resolved before "Accepted". The current "Recommended leaning" answers in Q1-Q4 are written as if they are the decisions (and indeed R8, R13, etc. are written that way). This creates a contradiction: are Q1 / Q2 / Q4 still open, or are they implicitly resolved? -> A reviewer cannot tell whether to treat the requirements as committed or provisional. -> Either move Q1 / Q2 / Q4 into "Decisions and Trade-offs" as committed (since the requirements text already commits to them) or mark which requirements are contingent on the open question.

## Suggested Improvements

1. **Make AC-X1 / AC-X2 reference Story 6 / Story 7 verbatim**: The Stories already contain example error messages with the exact line structure. Tie the AC golden assertions to those examples ("the rendered error message MUST match the structure shown in Story 6's diagnostic block, with at least the literal substrings X, Y, Z") so the contract is anchored to a visible reference rather than abstract line counts.

2. **Distinguish the two error contracts (substring vs. golden)**: R14 says messages must be testable via "golden-file or stderr-substring assertions". Pick one as the primary mechanism in this PRD and leave the other to design. Mixed messaging here will produce two divergent test styles in the resulting code.

3. **Clarify the `niwa migrate-source <name>` argument shape**: AC-M5 references "the literal workspace name" — confirm `<name>` is the workspace instance name (as registered) and not the source slug. The Stories use `my-workspace` consistently; lift that into R8 as the explicit definition.

4. **Tighten "first-fetch bandwidth" Known Limitation**: It says "may take noticeably longer than the resulting snapshot suggests" — a more concrete statement (e.g., "linear in repo size, not subpath size") helps reviewers evaluate the trade-off and avoids "noticeably" as a subjective term.

5. **Cross-link R5 to AC-D7 explicitly**: R5 commits to byte-identity; only AC-D7 verifies it. The other discovery-failure ACs (AC-D5, AC-D6) hint at it ("removed or unchanged") but do not formally check the contract. Either expand AC-D5 / AC-D6 to assert byte-identity, or add an AC-D8 covering ambiguity + no-marker byte-identity.

6. **Spell out the `niwa migrate-source` interactive prompt format**: AC-M4 says the command "asks the user to confirm via stdin (interactive mode)" but doesn't specify the prompt text or accepted responses. A passing test cannot be authored without this. Either commit to a prompt contract (e.g., "MUST print `Migrate <name> from <old-slug> to <new-slug>? [y/N]` and accept `y`/`yes` as confirmation") or defer to design with an explicit marker.

7. **Define behaviour when `--to` matches the current `source_url`**: `niwa migrate-source my-workspace --to acme/dot-niwa` (no-op) is unspecified. Should it exit zero silently, refuse, or print "already on this slug"? Worth a one-line clarification in R8.

## Summary

The PRD is unusually rigorous: it commits to single-call probe mechanics, names fixtures, ties ACs to specific requirements, and surfaces inherited limitations from upstream. The ambiguities found are mostly at the seams between requirements and their acceptance criteria (line-count assertions in AC-X1 / AC-X2, the "removed or unchanged" dual state in AC-D5, the no-`--to` non-interactive path in R8), and one structural mismatch where "Open Questions" overlap with already-committed requirements. None are large enough to block implementation; tightening them before `/design` will save a round of follow-up clarifications.

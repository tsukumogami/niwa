# Research: Infisical CLI admin surface for `niwa onboard`

Question driving this research: for each team-phase operation the `niwa onboard` wizard needs, can it delegate to the operator's own `infisical` CLI session, or must it fall back to REST-with-user-session-token or guided dashboard steps?

Local CLI checked: `infisical version 0.43.101` (installed at `~/.tsuku/tools/current/infisical`). Full `--help` tree walked for all top-level and relevant subcommands.

## Method

1. Ran `infisical --help` and every relevant subcommand `--help` locally (source of truth for "does this CLI build expose the verb at all").
2. Cross-checked gaps against official docs (infisical.com/docs) via WebSearch/WebFetch for REST endpoint paths, auth requirements, and plan gating.

## Key structural finding

The installed CLI (0.43.101) has **no `identity`, `org`, or `project` top-level command at all**. `infisical --help` top-level command list is exactly: `agent, bootstrap, cert-manager, dynamic-secrets, export, gateway, help, init, kmip, login, pam, proxy, relay, reset, run, scan, secrets, service-token, ssh, token, user, vault`. There is no command tree for organization or identity administration. `infisical identity --help`, `infisical org --help`, `infisical project --help` all fail with "unknown command." This means every "create/attach/grant" verb in the team-phase list is REST-or-dashboard by construction of the CLI itself — it isn't a matter of missing flags, the surface doesn't exist.

The only project-admin capability actually implemented in this CLI is folder management (`infisical secrets folders create|delete|get`), plus generic secret CRUD and service-token creation.

## Plan-gating note (identities generally)

Infisical's free tier caps total identities (human + machine) at 5, with paid tiers priced per additional identity (~$18/identity/month on Pro per one third-party teardown). This caps out organizations running solely on the free tier well before it's a "how do we automate this" problem — it's "you're out of identities." This is a hard external constraint on the wizard's team-phase flow regardless of what's automatable, and should be called out to the PRD/design as an assumption to validate against Infisical's current pricing page rather than trust this snapshot indefinitely.

## Consequences for the wizard's requirements

- **Nothing in the team-phase list (operations 1–3, 5) is CLI-delegable in the installed CLI version.** The wizard cannot shell out to `infisical identity create`, `infisical identity universal-auth attach`, `infisical identity client-secret create`, or `infisical project identity-membership grant` because none of those commands exist. Any "delegate to the operator's own CLI session" design for these four operations has to mean *use the CLI's stored session credential to drive raw REST calls*, not *invoke a CLI subcommand* — the CLI itself provides no verb surface here.
- Operation 4 (folder creation) is the one team-phase step that genuinely fits the "shell out to `infisical`" delegation model as literally stated — `infisical secrets folders create --path ... --name ...` does the job with no REST fallback needed.
- Operations 1, 2, 3, 5 all require either: (a) the wizard issuing REST calls itself using the bearer token already sitting in the operator's CLI session/keyring (feasible in principle — the CLI stores exactly this token and `infisical user get token` can surface it — but *whether that user-session JWT is accepted as bearer auth by the org/identity-management endpoints is not explicitly confirmed in the docs*; it's a reasonable architectural inference since the web dashboard is presumably a thin client over the same REST API and must be authenticating its own create-identity/attach-auth/grant-membership/mint-secret calls with the logged-in user's own session token, but this should be verified empirically — e.g. by capturing the dashboard's own network calls — before the design doc treats it as fact), or (b) degrading to guided dashboard instructions (Organization Settings > Access Control > Machine Identities, and Project Settings > Access Control > Machine Identities) for any operator who can't/won't have the wizard fire raw REST calls with their session token.
- Given the ambiguity on (a), the safer default for the PRD/design is: **treat operations 1, 2, 3, 5 as "REST-with-user-session-token, pending empirical confirmation" with a guided-dashboard fallback path**, and treat operation 4 as **CLI-native, no fallback needed**. Operation 6 (login/context/export/secrets set) is fully CLI-native and is the wizard's individual-phase backbone already validated separately.
- The unresolved technical risk worth flagging upstream: exact REST paths for "create client secret on identity" and "create project-identity-membership" were not pinned to a fetched, confirmed doc page in this pass (only inferred from convention/prose) — whoever writes the design doc should do one more direct fetch of infisical.com/docs/api-reference/endpoints/universal-auth/* and .../project-identities* before locking those calls into an implementation plan.

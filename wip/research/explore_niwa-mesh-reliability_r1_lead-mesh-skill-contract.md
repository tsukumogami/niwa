# Lead: What user-facing contract does the niwa-mesh skill describe today vs. what the runtime delivers, and what documentation needs to change alongside the fixes?

## Findings

### Skill file location and provisioning

The canonical skill content is generated by `buildSkillContent()` at
`internal/workspace/channels.go:682-833` and written to two filesystem
locations by `runPipeline` at `internal/workspace/channels.go:338-359`:

1. Instance-root: `<instance>/.claude/skills/niwa-mesh/SKILL.md`
   (channels.go:341).
2. Per-repo, for every non-coordinator role with a `repoPath`:
   `<repo>/.claude/skills/niwa-mesh/SKILL.md` (channels.go:354).

The two on-disk copies under `tsuku-3/.claude/skills/niwa-mesh/SKILL.md`
and `tsuku-3/public/niwa/.claude/skills/niwa-mesh/SKILL.md` are
byte-identical (verified: `diff` prints nothing) and match the generator
output. Only one canonical contract exists today.

The other eight `niwa-mesh/SKILL.md` files in the workspace tree (e.g.,
`public/shirabe/`, `public/dot-niwa/`, `public/tsuku/`, `public/koto/`,
`private/tools/`, `private/vision/`, `private/coding-tools/`) are the
per-repo mirrors written by Step 5 of the apply pipeline — they are also
identical-content and demonstrate that the skill is propagated into
every repo working tree.

### Inventory of runtime claims and verdicts

Each row quotes the skill verbatim with file:line citations against
`public/niwa/.claude/skills/niwa-mesh/SKILL.md` (the consumer-visible
copy; line numbers match all installed copies and the generator output
in `channels.go`).

#### a) Documented task state machine

Skill (frontmatter) at SKILL.md:7-8:

> "Tasks are first-class objects with a queued/running/terminal lifecycle
> owned by the niwa daemon."

Skill at SKILL.md:73-77:

> "Exiting the process without calling `niwa_finish_task` is classified
> as an unexpected exit and consumes a retry slot; after the cap is
> reached the daemon transitions the task to `abandoned` with `reason:
> "retry_cap_exceeded"`. A second finish call on a terminal task returns
> `TASK_ALREADY_TERMINAL`."

Runtime: `internal/mcp/types.go:171-189` defines exactly five states —
`queued`, `running`, `completed`, `abandoned`, `cancelled`. There is NO
`dangling` state in `validTaskStates`. The daemon's housekeeping moves
envelopes whose `task_id` filename matches no live `tasks/<id>/state.json`
into `<role>/inbox/dangling/<task-id>.json`
(`internal/cli/mesh_watch.go:786-802`), but `niwa_query_task` and
`niwa_list_outbound_tasks` continue to report the original `state` value
(typically `queued`).

Verdict: **broken by #112**. The skill's "queued/running/terminal"
lifecycle does not include the `dangling` filesystem classification that
the daemon emits. Coordinators see `state="queued"` for tasks that are
actually quarantined and unrecoverable through the documented `cancel`
path (which returns `too_late` per #112's evidence).

#### b) Documented ask/answer routing rules

Skill at SKILL.md:93-103:

> "If the target role has a live coordinator session, niwa routes the
> question directly to that session; otherwise it creates a task with
> `body.kind=\"ask\"` so the daemon can spawn a worker. A blocking
> coordinator receives questions via `niwa_await_task` returning
> `status:\"question_pending\"` with `ask_task_id` and `body`; a polling
> coordinator receives them via `niwa_check_messages` as
> `type==\"task.ask\"` messages. In both cases, answer by calling
> `niwa_finish_task(task_id=ask_task_id, outcome=\"completed\",
> result=...)`."

Runtime: `handleAsk` at `internal/mcp/server.go:780-843` does implement
the live-coordinator routing (lines 816-833 — when the worker has a
`mainInstanceRoot`, it looks up the live coordinator there, creates an
ask-task store entry, and writes a `task.ask` envelope to the
coordinator's inbox without a `task.delegate`). Tests for this exist at
`internal/mcp/session_registry_ask_test.go` (e.g.,
`TestHandleAsk_LiveCoordinator_WritesTaskAsk`,
`TestHandleAsk_SessionWorktreeRoutesToMainInstance`).

Verdict: **historically broken by #92, partly fixed**. Issue #92 is
still OPEN even though the live-coordinator routing now exists in code.
The skill correctly describes the intended behavior; the runtime now
matches it for the case where the worker is spawned inside a session
worktree (so `NIWA_MAIN_INSTANCE_ROOT` is set). What is NOT documented:
the entire mechanism only works when `mainInstanceRoot` is non-empty
AND a coordinator is registered live in `sessions.json`; otherwise
`handleAsk` returns the new `no_live_session` JSON (server.go:835-842).
The skill does not mention this `no_live_session` reply at all, so a
coordinator-less workspace falls into a path the agent has never been
told about.

#### c) Worker escalation (niwa_ask to coordinator)

Skill at SKILL.md:117-119:

> "Worker asks coordinator: inside a running task, call
> `niwa_ask(to=\"coordinator\", body=...)` with a tight timeout
> (60-120s) when you need clarification; fall back to
> `niwa_finish_task(outcome=\"abandoned\", reason=\"blocked: <detail>\")`
> if the ask times out."

Runtime: `niwa_ask` runs `s.isKnownRole(args.To)` at server.go:802. The
implementation at server.go:765-778 only checks
`<s.instanceRoot>/.niwa/roles/<role>/`. For a worker spawned inside a
session worktree, `instanceRoot` is the worktree, and
`scaffoldWorktreeNiwa` at `internal/mcp/handlers_session.go:80-108` only
provisions `roles/<repo>/` (line 86) — never `roles/coordinator/`.
Therefore `niwa_ask(to="coordinator")` from a worker hits
`UNKNOWN_ROLE` BEFORE the `mainInstanceRoot` fallback at server.go:817
ever runs. The same applies to `niwa_send_message` (server.go:714 calls
`isKnownRole` against the same local instance).

Verdict: **broken by #109**. The skill's primary recovery pattern is
unreachable. The fix at handleAsk:817 short-circuits to
`mainInstanceRoot` only AFTER `isKnownRole` has already returned false
locally. The check needs to also tolerate `to="coordinator"` when
`mainInstanceRoot != ""`, or `scaffoldWorktreeNiwa` must register a
synthetic `coordinator` role in the worktree's `.niwa/roles/`.

The "fall back to `niwa_finish_task(outcome=\"abandoned\")` if the ask
times out" advice is also misleading: in the current runtime the ask
fails synchronously with `UNKNOWN_ROLE` (no timeout to wait for), so the
documented sequence "tight timeout, then abandon on timeout" never plays
out as written.

#### d) Message vocabulary

Skill at SKILL.md:81-89:

> "Message `type` values follow the format
> `^[a-z][a-z0-9]*(\\.[a-z][a-z0-9]*)*$`. The daemon and MCP handlers
> emit: `task.progress`, `task.completed`, `task.abandoned`,
> `task.cancelled`. Agent-level peer exchanges use `question.ask`,
> `question.answer`, and `status.update`."

Runtime: every type in the "daemon and MCP handlers emit" list is
exercised — `task.progress`, `task.completed`, `task.abandoned`,
`task.cancelled` are all emitted by `internal/mcp/watcher.go:222-226`
and the report-progress handler. `task.delegate` and `task.ask` are
emitted (server.go:588, server.go:914, watcher.go:121) but **NOT
mentioned** in the skill's list — the skill omits the very vocabulary
that issues #92 and #109 hinge on. A coordinator polling
`niwa_check_messages` is told to expect `type=="task.ask"` later
(SKILL.md:99-100) but the Message Vocabulary section never declares
`task.ask` or `task.delegate` as part of the mesh's vocabulary.

Verdict: **incomplete documentation; not directly broken by an issue**.
The skill recommends `question.ask`/`question.answer` as the agent
peer-exchange types, but every actually-routed-by-the-runtime ask flows
through `task.ask`, not `question.ask`. No code path emits
`question.ask` today (`grep -n question\.ask internal/mcp/*.go` returns
only doc-comments and the format-regex example at server.go:242, 669).
A worker that took the skill literally and sent `question.ask` would be
silently ignored by the watcher (which only dispatches
`task.ask`/`task.completed`/`task.abandoned`/`task.cancelled`,
watcher.go:116-152).

#### e) Skills available in worker sessions

Skill (implicit via "Common Patterns") at SKILL.md:111-119: the worker
asks the coordinator only when blocked; the implication is the worker
has access to whatever skills it was instructed to run. The skill does
not enumerate which skills are or are not available in worker contexts.

The workspace-context.md `## Channels` section appended at
channels.go:856 reads:

> "See the `/niwa-mesh` skill for usage patterns."

— and the niwa-mesh skill is the only mesh-related skill made available
to workers (it is written into `<repo>/.claude/skills/niwa-mesh/`). No
`shirabe:*`, `tsukumogami:*`, or other workspace-level plugins are
copied or otherwise mounted into the worker spawn — confirmed by issue
#108's evidence (worker's `Skill` tool returns
`Unknown skill: shirabe:prd`).

Verdict: **broken by #108**. The skill never says "you have access to
shirabe/tsukumogami plugins" but coordinators write delegation bodies
that say things like `"Use /shirabe:prd"`, and the skill's own
`buildSkillContent` description says "common patterns such as
fan-out/collect or worker-asks-coordinator" — both depend on workers
being able to run the skills coordinators reference. The skill is
silent on this constraint, which is itself a documentation gap (the
skill should declare which plugin namespaces a worker is guaranteed to
have).

#### f) `niwa_create_session` behavior (referenced in skill's Session Management)

Skill at SKILL.md:144-147:

> "Use `niwa_create_session(repo, purpose)` to create a worktree and
> start its daemon; the call returns `{session_id, worktree_path}`."

Runtime: `handleCreateSession` at handlers_session.go:146-228 calls
`s.daemonStarter` (line 220) but `EnsureDaemonRunning` at
`internal/workspace/daemon.go:35-102` returns `nil` (success) even when
the daemon child died after `cmd.Start` succeeded — it polls for a PID
file for 500 ms and silently returns nil on timeout (lines 90-101). The
`daemon_warning` field in the response (handlers_session.go:224) only
fires on `cmd.Start` error, not on post-start child death (e.g.,
fsnotify exhaustion).

Verdict: **broken by #110**. The skill's wording "create a worktree and
start its daemon" is true at the syscall level (the daemon is forked)
but the daemon may die immediately afterward without surfacing the
failure to the caller. The skill makes no claim about how to verify the
daemon is healthy, and there is no documentation of the
`daemon_warning` field that does fire in the rare cmd.Start failure
case.

#### g) `niwa_list_sessions` (referenced in skill's Session Management)

Skill at SKILL.md:152-153:

> "Use `niwa_list_sessions` to rediscover active sessions after a
> restart (filter by `repo` or `status`)."

Runtime: `handleListSessions` at handlers_session.go:26-50 returns the
filtered `SessionLifecycleState` array — fields are `session_id`,
`repo`, `purpose`, `status`, `creation_time`, `worktree_path`,
`claude_conversation_id`, `parent_session_id` per
`docs/guides/sessions.md:77-86`. No `daemon` health subobject, no
`daemon_alive`/`last_claim_at` fields.

Verdict: **broken by #111**. The skill's "rediscover active sessions"
phrasing implies the listing is sufficient to know which sessions are
usable; in practice a `status: "active"` row may have a dead daemon
(see #110), and the skill never warns the caller that
`status="active"` is the lifecycle marker, not a runtime-health
indicator. `docs/guides/sessions.md:264-265` does mention "the state
file stays with `status: active`" when a daemon dies, but this nuance
is not reflected in the skill or in the API surface.

#### h) Body reuse / re-delegation

Skill at SKILL.md:120-123:

> "Cancel/update: while a task is still queued, the delegator can call
> `niwa_update_task` (returns `updated` if the task is still queued,
> `too_late` once it is running) or `niwa_cancel_task` (atomic rename
> into `inbox/cancelled/`)."

Runtime: matches — the documented cancel/update API exists. But the
skill does not describe any path for re-firing a body that abandoned or
dangled. The implicit recovery the skill leaves coordinators with is
"call `niwa_delegate` again with the original body." There is no
`niwa_redelegate` primitive documented or implemented.

Verdict: **gap acknowledged by #114**. Not strictly a contract break
(the skill does not promise re-delegation), but the skill should at
least note that bodies must be re-supplied verbatim today and that
re-delegation is a coordinator responsibility. With #112 (dangling
recovery), this becomes the only practical recovery path; the skill
should describe it.

#### i) Skill file pollution into consumer repos

Issue #97 is about the skill file *itself* leaking into delegated PRs.
The skill text says nothing about this — but the runtime
(channels.go:354) writes the skill into every consumer repo's working
tree where it gets staged by agents.

Verdict: **broken by #97**. Not a documentation contract break, but a
production bug whose fix likely changes how the skill is delivered to
worker sessions (e.g., outside the repo working tree), which then
*affects* the contract: the skill should specify where it lives so
agents do not commit it.

### Cross-reference: `docs/guides/sessions.md`

The sessions guide is more accurate than the skill in some places and
diverges in others.

- Daemon-may-die scenario: sessions.md:258-281 explicitly documents the
  case where the daemon dies after a session is active and explains
  manual recovery via `niwa session destroy`. This nuance is missing
  from the skill.
- `daemon_warning` field: sessions.md:222-225 documents this field on
  `niwa_create_session`. The skill does not.
- `branch_warning` field: sessions.md:107-110, 233-238 documents this
  on `niwa_destroy_session`. The skill mentions the branch-preservation
  semantics (SKILL.md:155-158) but not the structured `branch_warning`
  return field.
- The CLI surface (`niwa session list`, `niwa session destroy`,
  `niwa go`) is documented at sessions.md:132-205 but not referenced at
  all from the skill — agents driving the mesh through MCP may benefit
  from a pointer to the CLI for human-operated recovery.

### Documentation deltas required

The deltas split into (1) sentences to remove from the skill,
(2) sentences to amend, (3) sentences to add, and (4) cross-doc updates.

#### Sentences to amend (skill)

1. **Message Vocabulary section** at SKILL.md:81-89. Add `task.delegate`
   and `task.ask` to the "daemon and MCP handlers emit" list. Either
   demote `question.ask`/`question.answer`/`status.update` to "reserved
   for agent-defined peer exchanges (no built-in handler today)" or
   remove them entirely until the runtime actually dispatches them.
   Decision needed: the skill should not advertise vocabulary that no
   handler reacts to.

2. **Peer Interaction section** at SKILL.md:93-103. Add the
   `no_live_session` reply: when the target has no live coordinator AND
   the worker has no `NIWA_MAIN_INSTANCE_ROOT`, `niwa_ask` returns
   `{"status":"no_live_session", "role":"<target>", "message":...}`
   immediately without creating a task. The skill currently implies a
   spawn always happens; in reality the coordinator-target spawn path
   was removed and replaced by this synchronous reply
   (server.go:834-842).

3. **Common Patterns / Worker asks coordinator** at SKILL.md:117-119.
   Until #109 lands, the documented pattern is unreachable. Either:
   (a) prefix this paragraph with a known-issue note pointing at #109,
   or (b) hold the wording until the fix lands and the worker session's
   `.niwa/roles/coordinator/` is provisioned (or `isKnownRole` is
   updated). Today the call returns `UNKNOWN_ROLE` synchronously, so
   the "tight timeout (60-120s)" advice is wrong — there is no timeout,
   there is an immediate error.

4. **Session Management section** at SKILL.md:144-147. Replace "create
   a worktree and start its daemon" with a sentence that names the
   `daemon_warning` field and says daemon health is not a guarantee of
   the call returning success. Cross-link to sessions.md:258-281 for
   the dead-daemon recovery story.

5. **Session Management section** at SKILL.md:152-153. Amend the
   `niwa_list_sessions` description to clarify that `status` is the
   lifecycle marker (active/ended/abandoned), not daemon health, and
   that until #111 lands callers must check `daemon.log` or a future
   `daemon` subobject for runtime health.

#### Sentences to add (skill)

6. **A new state-machine subsection or dedicated paragraph** documenting
   the `dangling` classification. Suggested text:

   > "If a task envelope is queued but no daemon claims it within the
   > housekeeping window (the daemon failed to start, was killed, or
   > its session was destroyed), the daemon moves the envelope to
   > `<role>/inbox/dangling/<task-id>.json`. `niwa_query_task` continues
   > to report `state=\"queued\"` for these envelopes; the dangling
   > classification is currently a daemon-internal observable only
   > (see issue #112). Recovery: destroy the affected session, recreate
   > it, and re-`niwa_delegate` the body. There is no `niwa_resurrect`
   > or `niwa_redelegate` primitive today."

7. **A "Skills available in workers" subsection** documenting that
   workers spawned by `niwa_delegate` see a fixed Claude Code plugin
   set that today does NOT include `shirabe:*` or `tsukumogami:*`.
   Coordinators must not write delegations that mandate workspace-level
   plugins until #108 lands; instead, encode the structural
   instructions inline in the body.

8. **A pointer to `docs/guides/sessions.md`** at the top of the
   Session Management section. The skill currently re-derives session
   semantics; the guide is canonical and version-tracks better.

#### Sentences to remove (skill)

9. From SKILL.md:81-89, remove `question.ask`, `question.answer`, and
   `status.update` (or move to a "reserved" subsection). They are not
   wired to any handler and recommending them lures agents into a
   silent black hole.

10. From SKILL.md:93-99 routing rules: remove or rephrase "If the
    target role has a live coordinator session, niwa routes the
    question directly to that session; otherwise it creates a task
    with `body.kind=\"ask\"` so the daemon can spawn a worker." The
    "otherwise spawn a worker" clause was the original design but the

#### CLI / docs deltas

11. **`docs/guides/sessions.md`**: add a section on the `dangling`

12. **`niwa task list` / `niwa task show` output** (verify in

13. **`niwa session list` and `niwa mesh list` help text** (cli/mesh_list.go and the session list command in `internal/cli/session.go`):
    add a one-line note that `STATUS=active` does not imply a healthy

14. **Skill provisioning path itself** (per #97 fix): once the skill is

## Implications

- Five of the nine open issues represent direct breakages of skill
  contract claims (#92 partly fixed, #97, #108, #109, #110, #111, #112);
  one is a missing primitive that the skill assumes exists implicitly
  (#114); two are documentation gaps (parts of #109 and all of #113).
- The skill's "Worker asks coordinator" pattern is the most prominent
  user-facing claim that fails today — and it is the recovery path the
  skill itself directs agents to use when blocked. The combination of
  #108 (no plugins) plus #109 (cannot reach coordinator) means a worker
  that hits an unsatisfiable constraint has no documented escape hatch
  other than abandonment.
- The skill is not version-tracked separately from the runtime; it lives
  in the same repo as `channels.go` so amendments ship with code
  changes. This is good news for the design doc: the same PR that
  fixes any of #109/#110/#111/#112 should update `buildSkillContent`
  in lockstep, and a single CI check that diffs the rendered skill
  against a fixture would catch drift.

## Surprises

- Issue #92 ("niwa_ask spawns ephemeral worker instead of routing to
  active session") is technically already fixed in the source tree —
  `handleAsk` does write a `task.ask` notification to the live
  coordinator's inbox (server.go:823-833). The issue is still OPEN
  because the worker-side prerequisite (registered `coordinator` role
  in the worker session's `.niwa/roles/`) is not provisioned, so
  `isKnownRole` rejects the call before the routing logic runs. This
  ties #92 and #109 together — they are the two halves of the same
  end-to-end failure.
- The skill recommends `question.ask`/`question.answer`/`status.update`
  as the agent peer vocabulary, but no production code path emits any
  of these types. The watcher dispatches only on `task.*` types
  (watcher.go:121, 222-226). Agents that take the skill literally have
  always been silently ignored.
- `task.delegate` is the single most important type in the mesh (it is
  the type that the daemon uses to recognize a queued envelope —
  `daemonOwnsInboxFile` returns true iff `peek.Type == "task.delegate"`,
  mesh_watch.go:756) yet the skill's Message Vocabulary section never
  names it. This is a significant cognitive gap for any operator
  reading the skill to understand routing.
- The two installed skill copies (workspace-root and per-repo) are
  byte-identical. The architectural argument for two copies (per-repo
  visibility) is undermined by #97: each per-repo copy is a vector for
  pollution into PR diffs. A single workspace-root copy plus a
  `CLAUDE.local.md` reference might satisfy both delivery and
  isolation requirements.

## Open Questions

- Should the skill describe the runtime-internal `dangling` classification
  as a *visible* task state (with API fields) once #112 is fixed, or
  keep it as a daemon log signal and surface only the recovery primitive
  (`niwa_resurrect_task`)? The two answers imply different skill text.
- Is `question.ask`/`question.answer` ever going to be wired up, or
  should it be removed from the skill outright? If it is reserved for
  future use, that should be explicit; if it is dead vocabulary, deleting
  it removes a footgun.
- Once #109 lands, should the worker session's
  `<worktree>/.niwa/roles/coordinator/` be a real directory (matching
  the pattern used for other roles) or should `isKnownRole` grow a
  special-case for `to == "coordinator" && mainInstanceRoot != ""`?
  The skill text differs: option A means no skill change is needed
  beyond fixing the timeout description; option B requires the skill
  to explicitly describe that "coordinator" is a virtual route.
- Does the skill need its own changelog/version field? The mesh runtime
  evolves continuously; agents reading a stale on-disk skill (per #97
  the skill propagates into eight or more locations) may have
  conflicting versions. A `skill_version` in frontmatter plus a daemon
  check at session start would prevent drift but adds complexity.

## Summary

The niwa-mesh SKILL.md (generated by `internal/workspace/channels.go:682-833` and installed at instance-root plus every non-coordinator repo) makes nine first-class runtime claims; six of them are broken by open issues — task-state lifecycle omits `dangling` (#112), worker-asks-coordinator is unreachable because the worker's session worktree never registers a `coordinator` role for `isKnownRole` (#109), the message vocabulary section advertises `question.ask`/`question.answer`/`status.update` types no handler dispatches while omitting the actually-routed `task.delegate`/`task.ask` (compounds #92 and #109), `niwa_create_session` does not surface daemon-spawn failures (#110), `niwa_list_sessions` reports lifecycle status as if it were daemon health (#111), and worker sessions silently spawn without workspace plugins the skill's coordinator-fan-out patterns assume (#108); plus the skill file itself leaks into consumer repos via `channels.go:354` (#97). A coherent fix requires amending five existing skill paragraphs, removing the dead `question.*`/`status.update` vocabulary, adding a dangling-state paragraph and a worker-skills-availability paragraph, cross-linking `docs/guides/sessions.md`, and changing the skill delivery path so it stops being committed by delegated agents.

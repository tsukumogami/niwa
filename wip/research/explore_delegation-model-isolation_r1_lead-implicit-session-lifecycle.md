# Lead: Implicit session lifecycle design

## Findings

### How tasks are currently queued and routed

In `internal/mcp/handlers_task.go`, `handleDelegate` reads the optional `session_id` from
`delegateArgs` and passes it to `createTaskEnvelope`. That function calls
`resolveCreationInboxDir`, which bifurcates cleanly: a blank `session_id` routes the inbox
write to the main instance's role inbox (`<instanceRoot>/.niwa/roles/<role>/inbox`); a
non-blank `session_id` looks up the session's state file, validates status == "active",
path-guards the worktree against the main instance root, and writes to the worktree inbox
instead. The task's `state.json` records `session_id` so cancel and update can later
reconstruct the same worktree inbox path. No session ID means no worktree involvement.

### How sessions are created and destroyed

`handleCreateSession` in `internal/mcp/handlers_session.go` requires both `repo` and
`purpose` as inputs — `purpose` is validated non-empty before anything is written to disk.
There is no code path that generates a purpose string automatically. The session ID is
generated by `newSessionLifecycleID` as 8 random lowercase hex characters; neither the
task body nor any derived summary appears in the session ID or name.

`handleDestroySession` is idempotent on already-terminal sessions. For active sessions it:
kills in-progress worker processes (SIGKILL, via `killSessionWorkers`), writes
`status = "ended"` to the lifecycle file, stops the per-worktree daemon, runs
`git worktree remove --force`, and attempts `git branch -d` (or `-D` with `force=true`).
When the branch has unmerged commits and `force=false`, the worktree directory is gone but
the branch is left in place, with `branch_warning` returned in the response. There is no
auto-destroy on task completion — destroy is always explicit.

### The session lifecycle schema

`SessionLifecycleState` in `internal/mcp/session_lifecycle.go` has three status values:
`active`, `ended`, `abandoned`. There is no `ephemeral` flag, no `auto_destroy` field, and
no TTL or task-count field. The schema also lacks any field linking a session to a
particular task ID at creation time — sessions and tasks are correlated only at delegation
time via `session_id` stored in `TaskState`. There is no concept of a "single-task mode"
in the current schema.

### The PRD's "no implicit sessions" decision

The PRD (`docs/prds/PRD-mesh-session-lifecycle.md`, Decisions section) states:

> **No implicit sessions on untagged niwa_delegate.** A delegation without a `session_id`
> continues to behave exactly as today. This preserves backward compatibility and makes the
> session model strictly opt-in. Implicit sessions were rejected because they would
> silently change the behavior of all existing coordinator prompts.

The anticipated failure mode was silent behavioral change: every existing coordinator
prompt that calls `niwa_delegate` without a `session_id` would start creating worktrees,
git branches, and daemons. Coordinators not expecting isolation would accumulate session
debris without any indication. The PRD framed this as a backward-compatibility problem, not
a semantics problem.

### Auto-generated purpose string

No code in the current codebase generates a purpose string. `handleCreateSession` rejects
empty `purpose` with `BAD_PAYLOAD`. An implicit session would need to derive a purpose from
the task body. The natural candidate is the first N characters of the task `body` field
(e.g., truncated to 256 UTF-8 characters matching the R21 limit). `bodySummary` in
`handlers_task.go` already collapses and truncates body JSON to 200 characters for list
output, so the pattern exists. However, task bodies are opaque JSON blobs — they may be
deeply nested objects with no human-readable top-level string. An auto-derived purpose would
often be machine-generated JSON noise, not meaningful to a human inspecting
`niwa session list`. The PRD treats purpose as "the sole human-readable label for a session
at the MCP level" (R24), making a generated purpose semantically weaker than the current
model intends.

### Auto-destroy after task completion: trade-offs

The current destroy flow does four things with side effects: kills workers, writes lifecycle
file, stops daemon, removes worktree and branch. An auto-destroy triggered by task
completion would need to be initiated by the daemon after it writes the terminal state
transition — which happens in `UpdateState` called from `handleFinishTask`. The daemon
currently has no hook that calls back into session lifecycle after a task goes terminal.

Trade-offs of auto-destroy:

- **Pro**: no leaked worktrees or branches; workspace stays clean without coordinator action.
- **Con**: if the worker completes but the result has not been read by the coordinator,
  destroying the worktree immediately removes the working tree files that the coordinator
  may need to inspect or merge. Branch deletion before review discards history.
- **Con**: task failure mid-execution leaves the worktree in a partial state. The current
  `killSessionWorkers` path in `handleDestroySession` handles SIGKILL cleanup, but a task
  that calls `niwa_finish_task(outcome=abandoned)` would trigger auto-destroy while the
  branch may have partial commits. With `force=false` branch deletion, the branch would be
  preserved (unmerged), but the worktree would still be removed.
- **Con**: the current lifecycle schema has no `task_id` → `session_id` reverse mapping at
  the session level. The daemon would need to look up the session ID from the task's
  `state.json` and then call back into session destroy logic, adding coupling between the
  task and session code paths that doesn't exist today.

### Branch fate on auto-destroy

`handleDestroySession` with `force=false` runs `git branch -d` — which only succeeds when
the branch is already merged. For an ephemeral task session, the branch would rarely be
merged at completion time (the typical lifecycle ends with the worker pushing and the
coordinator reviewing, not an immediate merge). So auto-destroy with `force=false` would
consistently produce `branch_warning` responses and leave orphaned branches. Auto-destroy
with `force=true` would silently delete unmerged work, which contradicts the PRD's stated
goal of preventing accidental data loss (R4, R5).

### Task failure mid-execution

If the worker calls `niwa_finish_task(outcome=abandoned)` and auto-destroy fires, the
worktree removal still occurs (the daemon would have to call the destroy path). Any partial
commits on the `session/<id>` branch survive branch deletion unless `force=true`. If the
worker process crashes (unexpected exit), the daemon retries up to `max_restarts` times;
the session worktree is not touched during retries. An auto-destroy on failure would fire
only after all retries are exhausted — but the current daemon has no such callback.

### Session persist with explicit coordinator destroy

Under the current model, the coordinator is always responsible for calling
`niwa_destroy_session`. There is no auto-expiry. This was deliberate: "No auto-pruning of
orphaned sessions" (PRD Decisions). The consequence for implicit sessions is that the
coordinator must either be told to destroy the session after task completion (via a new
signal or by convention) or sessions accumulate until explicit cleanup. The PRD's R9
says orphaned sessions surface in `niwa_list_sessions` with a stale marker — an implicit
session whose coordinator context has reset would appear as stale with no record of why it
was created.

### Ephemeral worktree vs. persistent session: are these two distinct concepts?

The current session model is explicitly designed for multi-task continuity (PRD goals 1–2,
R2, R10, R11): a session is a container for a conversation that spans multiple sequential
tasks. The `ClaudeConversationID` field in `SessionLifecycleState` and the R10/R11
requirements for `--resume` are the core value proposition. An ephemeral single-task
session is structurally different: it exists only to provide worktree isolation for one
task, not to preserve conversation continuity across tasks. There is no conversation to
resume and no prior worker to inherit context from.

These are architecturally distinct:

| Attribute | Persistent session (current) | Ephemeral per-task session |
|---|---|---|
| Lifetime | Multiple tasks, coordinator-bounded | Single task, task-bounded |
| Conversation continuity | Yes (`--resume`) | No (or irrelevant) |
| Branch fate | User decides at destroy time | Must be decided automatically |
| Purpose string | Human-supplied | Derived or synthetic |
| Owner/destroyer | Coordinator explicit | Auto (daemon?) |
| Schema support | Full (`active/ended/abandoned`) | Requires new mode or separate type |

The current schema and lifecycle code do not support ephemeral single-task mode. Fitting
ephemeral semantics into the existing `SessionLifecycleState` would require at least an
`auto_destroy` flag and a daemon callback on task terminal transition. Alternatively, a
separate lightweight "task worktree" concept could be designed that shares the git worktree
provisioning code but has simpler lifecycle semantics (no conversation ID, no purpose
string, auto-cleanup by daemon).

### Does the current lifecycle support ephemeral single-task mode?

No. The schema has no `auto_destroy`, `task_id`, or `ttl` field. The daemon has no
post-task-terminal callback into session destroy logic. `handleDestroySession` is an MCP
tool, not an internal daemon hook. Implementing implicit sessions would require:

1. A new field or flag on `SessionLifecycleState` (e.g., `ephemeral: true`) to mark
   auto-destroy intent.
2. A daemon hook that fires when a task transitions to a terminal state and, if the task's
   `session_id` points to an ephemeral session, triggers worktree cleanup.
3. A decision on branch fate for ephemeral sessions with failed tasks (auto-delete with
   `force` vs. preserve orphan branch).
4. A purpose generation strategy that is meaningful in `niwa session list` output.

## Implications

**Implicit sessions are not a free add-on.** The existing session lifecycle is designed
around explicit coordinator control. Every implicit session is a worktree + branch +
daemon + lifecycle file created automatically; without a coordinator-side destroy call,
these accumulate. The PRD's backward-compatibility decision was made with this operational
cost in mind.

**The PRD's rejection was about silent behavioral change, not semantics.** The concern was
that all existing untagged `niwa_delegate` calls would start producing worktrees without
coordinators knowing. A narrower design (only new calls opt in, or only calls that
explicitly request isolation) avoids that failure mode while still allowing worktree
isolation per task.

**Ephemeral worktrees are a separate design problem.** The value of per-task worktree
isolation (preventing dirty main clone, enabling parallel work, preventing branch conflict)
is real and independent of multi-task session continuity. These two problems may deserve
separate design surface — one for "give this task a clean worktree" and one for "group
these tasks into a continuing conversation."

**Branch ownership is the hardest constraint.** Any auto-destroy design must answer: what
happens to the branch when the task fails mid-way? The current `git branch -d` guard
against unmerged commits exists for exactly this case. An ephemeral session design must
either accept potential branch orphans, accept `force=true` deletions (data loss risk), or
define a new branch lifecycle that differs from the explicit session model.

**The coordinator still needs to know sessions exist.** Even with auto-destroy, the
coordinator's context window may not survive long enough to learn that a session was
created and destroyed. If the coordinator calls `niwa_list_sessions` after context reset,
it may find no trace of implicit sessions that already cleaned up. Session auditability
degrades.

## Surprises

**`purpose` is required with no default.** There is no fallback code and no auto-generation
path. This is a harder blocker than it might appear: the PRD explicitly makes purpose "the
sole human-readable label for a session at the MCP level," so a meaningless auto-generated
purpose defeats the intent.

**`handleDestroySession` is only an MCP tool, not an internal function.** There is no
`destroySession()` internal function that the daemon could call. To support auto-destroy,
the daemon would need to call into the same worktree-removal and git-cleanup logic, which
would require refactoring `handleDestroySession` into a callable utility.

**The branch warning is returned in the MCP response, not written to disk.** `BranchWarning`
in `SessionLifecycleState` has `json:"branch_warning,omitempty"` but the comment says
"Never written to disk; only present in the MCP/CLI response." An auto-destroy that fails
to clean up a branch would produce a warning that no one is listening to — coordinators
in auto-destroy mode receive no MCP response.

**The "no implicit sessions" decision was about backward compatibility specifically.** The
PRD did not say implicit sessions are architecturally wrong; it said they would break
existing coordinators silently. This means the design space remains open for an opt-in
implicit session mechanism that doesn't affect untagged `niwa_delegate` calls.

## Open Questions

1. **What is the right design surface?** Should implicit isolation be modeled as a
   lightweight "task worktree" (separate from session lifecycle) or as an ephemeral session
   mode within the existing lifecycle? The former avoids schema coupling; the latter reuses
   existing infrastructure.

2. **Who generates the purpose string?** If the calling coordinator generates it (e.g., a
   short label in the `niwa_delegate` call body), the API is no longer implicit — it
   requires the coordinator to pass extra context. If niwa generates it, `niwa session list`
   becomes harder to read.

3. **What happens to the branch on task failure?** This is the key data-loss trade-off.
   Require human input before deleting unmerged branches, or accept auto-deletion of
   in-progress work?

4. **Should auto-destroy fire on task completion, task terminal state, or session idle
   timeout?** Completion and abandonment have different implications for branch state.
   An idle timeout (session with no running tasks for N minutes) is a weaker trigger but
   gives humans a window to inspect work.

5. **Does the coordinator need to observe implicit session creation?** If implicit sessions
   are transparent to the coordinator, it cannot list them, query them, or destroy them
   manually if auto-destroy fails. Is that acceptable?

6. **What happens if the daemon crashes between task completion and auto-destroy?** The
   session would be left in `active` state with a stale creator PID and no task still
   running. Does it appear as orphaned? Who cleans it up?

7. **For the human-in-the-main-clone UX**: the core question (what's the right UX for
   humans working directly in the main clone) may not require implicit sessions at all. If
   humans are always creating sessions explicitly via `niwa session create`, the coordinator
   opt-in session model is already sufficient for them. The implicit session question is
   specifically about coordinator-managed delegation workflows, not human CLI workflows.

---

## Summary

The PRD explicitly rejected implicit sessions to preserve backward compatibility, not
because the design is semantically unsound — every existing untagged `niwa_delegate` call
would silently start creating worktrees, daemons, and branches. The current session lifecycle
schema and daemon have no support for ephemeral single-task mode: there is no auto-destroy
flag, no daemon hook that fires on task completion, and `handleDestroySession` is an MCP
tool rather than an internal callable. The biggest open question is branch fate: any
implicit session design must decide what happens to a `session/<id>` branch when the task
fails mid-way, since the existing guard against unmerged-branch deletion will consistently
block automatic cleanup for in-progress work.

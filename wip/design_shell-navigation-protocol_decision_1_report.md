### Decision: Protocol Mechanism

**Context**

Two niwa commands — `create` and `go` — are "cd-eligible": after they complete, the shell should change to a directory the CLI determined. Because a subprocess can't change its parent shell's working directory directly, a side-channel protocol is required.

The current protocol captures all stdout with `$()` and treats the entire output as the directory path. This breaks silently whenever anything else writes to stdout — git clone progress, fmt.Printf diagnostic lines, or third-party library output. Future output modes (--verbose, --debug, --json, NIWA_LOG_LEVEL=debug, CI quiet modes) will make this collision surface larger over time.

The shell wrapper lives in `~/.niwa/env`, generated by `shellWrapperTemplate` in `internal/cli/shell_init.go`. It is installed by `niwa shell-init install` and sourced at shell startup. Updating it requires users to re-run the install command — the wrapper should not need frequent changes, and any new protocol must remain parseable in POSIX bash/zsh without external tools.

**Assumptions**

- The CLI can write to any file descriptor or path it chooses; the constraint is on the shell wrapper reading that output.
- fd 3 is not reliably available as a free descriptor in all shell contexts — some shells or wrapper layers may have it open.
- Temp file creation (`mktemp`) is available in all target environments (bash/zsh on macOS and Linux).
- Users run `niwa shell-init install` once during setup; a one-time re-install to migrate the wrapper is acceptable, but must be explicitly documented.
- The CLI controls its own direct stdout writes; it cannot intercept stdout from subprocesses that inherit the fd.

**Chosen: Option D — Temp file via environment variable**

Before invoking the niwa binary, the shell wrapper creates a temp file with `mktemp` and exports its path in `NIWA_RESPONSE_FILE`. The CLI detects this variable and, on success, writes only the landing path to that file. The wrapper reads the file after the command exits, calls `cd`, and removes the file. Stdout and stderr flow unmodified to the terminal throughout.

**Rationale**

The core problem is that stdout and stderr are shared streams — any output on them, from any source, can corrupt a protocol that relies on those streams being clean. Options A and B both require the shell wrapper to parse and filter a stream that the CLI cannot fully control (third-party subprocesses write wherever they write). Option C uses fd 3 as a private channel, which cleanly avoids the stream-pollution problem, but assumes fd 3 is free — an assumption that breaks silently in nested shell environments, Docker entrypoints, or tools that happen to open fd 3.

Option D uses a temp file as the channel. No shared stream is involved. The CLI writes to a path it received from the wrapper; the wrapper reads from a path it created. Stdout and stderr are never touched by the protocol. The only external dependency is `mktemp`, which is universally available. Failure modes are explicit: if `mktemp` fails, the wrapper logs a warning and falls back to running niwa without cd-navigation; if the CLI doesn't write the file (non-cd-eligible exit path), the wrapper reads nothing and skips the `cd`. Neither failure is silent or data-corrupting.

The wrapper complexity is low: three lines of setup before the call, three lines of cleanup after. The logic is straightforward enough for non-shell-experts to read and audit. The protocol is also future-proof: adding --json output, --debug logging, or CI quiet modes requires no wrapper changes, because those modes write to stdout/stderr which the protocol never touches.

**Alternatives Considered**

- **Option A (stdout sentinel line)**: The wrapper reads niwa's stdout line by line, filters out the `NIWA_CD=...` line, and passes remaining lines to the terminal. This is appealing because it requires no external state, but it fundamentally can't work reliably. The CLI cannot redirect stdout from third-party subprocesses — git, gh CLI, and other tools write directly to fd 1. The wrapper would need to buffer all stdout before printing it, adding latency to every create/go invocation. Any subprocess that writes a line matching the sentinel pattern (unlikely but possible) would be silently consumed. Shell loop-based line filtering in POSIX bash/zsh is also fragile around binary output and large buffers. Rejected because the stream-isolation problem cannot be solved at the wrapper layer.

- **Option B (stderr sentinel line)**: Same structure as Option A but on stderr. Stderr is less commonly piped by users, but it's not a private channel — git writes progress to stderr, the CLI already writes diagnostic messages there (`go: workspace "foo"\n`), and any future --verbose mode would write there too. The filtering logic is the same as Option A and carries the same fragility. One marginal advantage: CI pipelines that redirect `2>/dev/null` would silently drop the sentinel, making the failure mode cleaner than Option A (no cd rather than wrong cd). But the fundamental problem — parsing a stream shared with uncontrolled subprocess output — is identical. Rejected for the same reason as Option A, with the added concern that stderr is already used for user-facing diagnostic output in the current codebase.

- **Option C (dedicated file descriptor)**: The wrapper opens fd 3 with `exec 3>/tmp/niwa-fd-XXXX` before invoking niwa, and niwa writes to `/dev/fd/3`. Stdout and stderr flow normally. This is conceptually clean — fd 3 is a private channel between wrapper and binary. The problem is availability: fd 3 may already be open in certain environments. Shells invoked from editors (VS Code integrated terminal), CI runners (GitHub Actions), Docker containers, and some multiplexers open additional fds. The Go runtime also opens fds during startup. If fd 3 is taken, the write either corrupts an unrelated open file or fails with EBADF — both silent from the user's perspective. Choosing a higher fd number (e.g., fd 9) reduces collision probability but doesn't eliminate it, and adds complexity to both the Go code (`os.NewFile`) and the wrapper (`exec 9>...`). The temp-file approach delivers the same stream isolation without the fd availability assumption. Rejected because it trades one fragility (stream pollution) for another (fd availability) rather than eliminating the class of problem.

**Consequences**

- The shell wrapper must be updated: `niwa shell-init install` must be re-run once to deploy the new wrapper. Users on the old wrapper continue to get the broken stdout-capture behavior until they upgrade; this should be documented in the release notes.
- The CLI gains a new code path: when `NIWA_RESPONSE_FILE` is set, write the landing path to that file instead of (or in addition to) stdout. The stdout write can be removed for cd-eligible commands, or retained for non-interactive use (`NIWA_RESPONSE_FILE` absent means the caller wants stdout, e.g., scripting).
- The protocol is opt-in from the CLI's perspective: if `NIWA_RESPONSE_FILE` is not set, the command behaves as before, which preserves backward compatibility for scripts that invoke `niwa go` directly and capture stdout.
- Temp file cleanup must be robust: the wrapper should remove the file in both success and failure paths. A `trap` or explicit cleanup after the command call handles this.
- The sentinel format (Decision 2) becomes a non-issue for the file channel: the file contains only the path, with no need to distinguish it from other output.

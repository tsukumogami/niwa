# Decision 4: Bootstrap Scaffold Shape and Derivation

**Question:** What exactly does the bootstrap scaffold emit on disk, and
how are derived values populated?

**Tier:** Standard (Tier 3, fast path: phases 0, 1, 2, 6).

**Mode:** `--auto`.

## Context

The exploration produced a "minimal-ideal" scaffold proposal
(`wip/research/explore_init-bootstrap-empty-source_r1_lead-minimal-scaffold.md`)
with several open details. This decision picks the concrete shape that
`workspace.Scaffold` will emit on the bootstrap path
(`niwa init <name> --from <empty-remote> --bootstrap`).

Inputs available at the bootstrap call site:

- Workspace `name` (positional arg, defaultable to `"workspace"`).
- `--from` slug parsed into `(host, org, repo, subpath, ref)`. The org is
  always present.
- A `github.APIClient` constructed with `resolveGitHubToken()`. Today the
  client has `ListRepos` (org-scoped) and `HeadCommit` / `FetchTarball`
  (single-repo); it does **not** yet have a single-repo metadata fetch
  for `GET /repos/{owner}/{repo}`.

Existing surfaces the decision must respect:

- The current `workspace.Scaffold(dir, name)` signature is consumed from
  two call sites (`modeScaffold`, `modeNamed`) in `internal/cli/init.go`.
  Backward-compatibility for those callers matters.
- Niwa's idiom for informational messages at init time is stderr `note:`
  lines (see `emitVaultBootstrapPointer`, name-override note).
- `dot-niwa/.niwa/workspace.toml` is the only public reference and emits
  active `name` + `content_dir` + one `[[sources]]` + one
  `[groups.<vis>]`. It omits `default_branch`.

## Alternatives Considered

### S-A — Exploration proposal (derived active, no extras)

Active `[workspace]` (name, content_dir). Active `[[sources]]` from the
`--from` org. Active `[groups.<vis>]` from `GET /repos/{owner}/{repo}`
`private` bool. Commented `[claude.content.workspace]`. One schema doc
link. On lookup failure → default to `[groups.public]`. No `.gitkeep`.
No stub `workspace.md`. No audit comment. `workspace.Scaffold` gains a
struct param.

### S-B — Conservative (commented placeholders only)

name + content_dir active. All other sections commented placeholders.
No GitHub API call. User edits the file in the worktree before push.

### S-C — Aggressive (S-A plus stubs and audit comment)

S-A plus `.niwa/claude/.gitkeep`, stub `.niwa/claude/workspace.md`,
and `# Bootstrapped by niwa from <slug> on <date>` audit comment at the
top of the file.

### S-D — Mid-aggressive (S-A plus `.gitkeep`)

S-A plus `.niwa/claude/.gitkeep` (so the directory pushes cleanly).
No stub `workspace.md`. No audit comment.

## Evaluation

### Apply ergonomics

S-A and S-D produce a workspace.toml that `niwa apply` will succeed on
without user edits (because `[[sources]]` and `[groups.*]` are active).
The minimum-friction goal of the feature — "user opens worktree, sees
sensible scaffold, decides whether to push" — is best served by a
configuration that *already works* rather than one that demands editing.

S-B punts to the user: every bootstrap call would still require manual
editing of the placeholder before `niwa apply` can do anything. That
defeats the value the user explicitly asked for ("the minimal ideal
scaffold proposed by niwa").

### Worktree push hygiene

`git add .niwa/` silently drops empty directories. Today's `Scaffold`
creates `.niwa/claude/` empty; in the existing init flows that's fine
because nothing pushes from `Scaffold`'s output. The bootstrap path
*does* push (or, more precisely, stages and commits before handing the
worktree to the user). If the user later adds `[claude.content.*]` entries
referencing `claude/...`, the convention only works if the directory
exists in the pushed tree.

`.gitkeep` is the standard zero-content convention. Adding it has no
runtime cost and removes a footgun. S-D and S-C both include it; S-A
omits it. The exploration's lead-minimal-scaffold explicitly flagged
this as Open Question 6 with the observation "if the empty-source flow
auto-stages, a `.gitkeep` matters" — the bootstrap flow *does*
auto-stage by design.

### Stub `workspace.md` (S-C only)

A pre-created `.niwa/claude/workspace.md` would be discoverable by `ls`,
but it is also the kind of empty file that triggers
"did-you-forget-to-fill-this-in?" friction. The exploration explicitly
rejected this stub for the same reason. Including it also requires
deciding what placeholder content to write — itself a decision point
that has no good default. Reject.

### Audit comment (S-C only)

`# Bootstrapped by niwa from <slug> on <date>` provides provenance for
a future reader. Trade-offs:

- Pro: when this file lands on a teammate's machine, they can answer
  "where did this come from?" without spelunking. The pattern matches
  generators in adjacent ecosystems (`# Generated by ...`).
- Con: the comment is durable in the team config repo (it survives
  forever in `dangazineu/commuter`'s `.niwa/workspace.toml`). Dating it
  is awkward — the date when *this* user bootstrapped has no meaning to
  a teammate who clones months later. The slug embeds an org/repo that
  the workspace already implicitly references (via the parent repo
  URL), so it's not novel information.
- The minimal-ideal goal points away from boilerplate. dot-niwa carries
  no such header.

On balance: an audit comment that's *informationally durable* (origin)
adds dubious value; one that's *transient* (timestamp) becomes noise.
Reject for both S-A and S-D. (If a user wants provenance they can
write it themselves in their first commit message — which `niwa
bootstrap` already produces.)

### Visibility lookup mechanics

`GET /repos/{owner}/{repo}` returns `private` bool. The existing
`github.APIClient` does not expose this; the design will need a new
method (call it `GetRepo(ctx, owner, repo)`) or an inline call. Either
way, the lookup is one HTTP request, cheap.

Failure modes:

- **No network**: lookup errors. Treat as soft failure.
- **No token, private repo**: 404. The repo's mere existence couldn't
  be probed, but `--from` already resolved the clone URL successfully,
  so the user has *some* form of access (SSH key, etc.). Still soft
  failure on the API path.
- **Token without read access**: 404, same handling.
- **404 missing repo**: bootstrap path is only entered after the
  `NoMarkerError` from a tarball fetch that *succeeded* — so the repo
  exists. A 404 from `repos/get` while the tarball succeeded would
  indicate API skew or a token-scope mismatch; same handling.

The right fallback is `[groups.public]`. Rationale:

1. Public is the more permissive classification — a workspace later
   discovered to be private gets reclassified by the user; the reverse
   (private repos accidentally public-grouped) is also user-fixable on
   first apply. Either direction is reversible by a one-line edit.
2. Public has no auth requirement, so applying with a single
   `[groups.public]` block "just works" for the most common open-source
   case.
3. The alternative — emitting *no* `[groups.*]` block on failure —
   makes the resulting workspace.toml a no-op on `niwa apply` (groups
   are how sources get classified to disk). That breaks the
   "scaffold-then-apply-works" property the user asked for.

A stderr `note:` line when the lookup fails (consistent with niwa's
existing pattern) tells the user what happened:

```
note: could not determine remote visibility (network|auth); defaulting to [groups.public]. Edit .niwa/workspace.toml to change.
```

### `workspace.Scaffold` signature

Today: `Scaffold(dir, name string) error`. Two call sites pass
positional args.

The bootstrap path needs at least `Org` and `Visibility` (plus possibly
`SourceSlug` for future audit-comment opt-ins, though we reject that
above). Three approaches:

1. **Add a struct param to `Scaffold`**: change all callers, even
   `modeScaffold`/`modeNamed` which don't need the extras.
2. **Introduce a sibling function** `ScaffoldFromSource(dir string,
   opts ScaffoldOptions) error` for the bootstrap path; keep
   `Scaffold(dir, name string) error` for the existing two callers.
3. **Add an optional `ScaffoldOptions` overload** with variadic
   functional options.

Pick (2). Rationale:

- The existing callers are correct as-is and don't benefit from a
  wider signature.
- The bootstrap path has structurally different inputs (it requires
  an org; it may have a visibility) and a different output template
  (active vs commented sections). Co-locating them in one function
  forces conditional rendering inside the template, which is harder
  to test.
- A sibling function lets us share the *parts* of the template that
  are identical (the schema doc-link footer, the commented
  `[claude.content.workspace]` hint) via a small helper, without
  forcing one function to serve two intents.

### Sources: active vs commented

Active. The org came from the user's own `--from` argument; this is
not a guess. The 90% case is "manage repos in the same org as the
config repo," and the user can add more sources by appending another
`[[sources]]` block. If the user's intent is multi-org, they edit the
file in the worktree before pushing — the same way they'd edit any
other field.

Commented `# org = "<org>"` would force every user, even the
single-org majority, to uncomment before apply works. That's the
friction we're removing.

### Reference: literal proposed scaffold

For `niwa init <name> --from <org>/<repo> --bootstrap` where
`<org>/<repo>` is `dangazineu/commuter` and `repos/get` reports
`"private": false`:

```toml
[workspace]
name = "<name>"
content_dir = "claude"

[[sources]]
org = "<org-from-slug>"

[groups.<public|private>]
visibility = "<public|private>"

# CLAUDE.md content hierarchy: drop a workspace.md in .niwa/claude/ to populate.
# [claude.content.workspace]
# source = "workspace.md"

# See https://github.com/tsukumogami/niwa/blob/main/docs/guides/workspace-config-sources.md
# for the full schema (claude.*, env.*, vault.*, files, channels, instance).
```

Concretely for `dangazineu/commuter` with a public repo:

```toml
[workspace]
name = "commuter"
content_dir = "claude"

[[sources]]
org = "dangazineu"

[groups.public]
visibility = "public"

# CLAUDE.md content hierarchy: drop a workspace.md in .niwa/claude/ to populate.
# [claude.content.workspace]
# source = "workspace.md"

# See https://github.com/tsukumogami/niwa/blob/main/docs/guides/workspace-config-sources.md
# for the full schema (claude.*, env.*, vault.*, files, channels, instance).
```

Plus on-disk:

- `.niwa/workspace.toml` — the file above.
- `.niwa/claude/.gitkeep` — empty file; ensures the directory pushes
  cleanly so `[claude.content.workspace] source = "workspace.md"`
  (when the user uncomments) resolves against an existing path.

No stub `workspace.md`. No audit-trail comment. No `default_branch`.
No `# version` comment.

## Decision

**Chosen: S-D (S-A plus `.niwa/claude/.gitkeep`).**

`workspace.Scaffold` stays as-is; the bootstrap path calls a new sibling
`workspace.ScaffoldFromSource(dir string, opts ScaffoldOptions) error`
where `ScaffoldOptions` carries `Name`, `Org`, `Visibility` (string,
empty when lookup failed), and `IncludeGitkeep` (always true for
bootstrap, parameterized so unit tests can disable it).

### Concrete shape

- Active `[workspace]`: `name`, `content_dir = "claude"`. Drop
  `default_branch` and the `# version = "0.1.0"` comment.
- Active `[[sources]]` with `org = "<org-from-slug>"`.
- Active `[groups.<vis>]` with `visibility = "<vis>"`. When the
  visibility lookup fails (network, auth, 404), default to
  `[groups.public]` and emit a stderr `note:` explaining the fallback.
- Commented `[claude.content.workspace] source = "workspace.md"` as a
  convention hint.
- Single schema-doc link replaces all other commented examples.
- `.niwa/claude/.gitkeep` created empty so the directory pushes cleanly.
- No stub `workspace.md`. No audit-trail comment.

### Implementation notes

- Add `(*APIClient).GetRepo(ctx, owner, repo) (*Repo, error)` in
  `internal/github/client.go`, returning the existing `Repo` struct
  (uses `private` bool → `Visibility` normalization the same way
  `ListRepos` already does).
- Soft-fail visibility lookup: any error → empty `Visibility`, scaffold
  emits `[groups.public]`, stderr note explains.
- Sibling function in `internal/workspace/scaffold.go`:

  ```go
  type ScaffoldOptions struct {
      Name           string
      Org            string
      Visibility     string // "public" | "private" | ""
      IncludeGitkeep bool
  }

  func ScaffoldFromSource(dir string, opts ScaffoldOptions) error
  ```

- Both `Scaffold` and `ScaffoldFromSource` create `.niwa/claude/`. Only
  `ScaffoldFromSource` writes `.gitkeep` (gated on
  `opts.IncludeGitkeep`).

## Confidence

**Medium–high.** The dot-niwa reference, the exploration's deep lead
research, and niwa's existing stderr-`note:` idiom converge on this
shape. The two genuinely contested sub-points (`.gitkeep` yes/no;
signature reshape) have clear-tiebreaker arguments rather than 50/50
calls. The audit-comment rejection has the weakest supporting argument
("noise vs provenance") and is the most reversible choice — if user
feedback wants it, S-D plus the comment is a one-line addition.

## Sub-Question Answers

| Sub-question | Answer |
|---|---|
| `[[sources]] org` active or commented? | **Active**. The slug came from the user. |
| Query GitHub for visibility? | **Yes**, via a new `GetRepo` API. |
| Fallback on visibility lookup failure? | **`[groups.public]`** + stderr `note:`. Do not skip the group entirely. |
| Pre-create `.niwa/claude/.gitkeep`? | **Yes**. The bootstrap flow auto-stages; empty dirs get silently dropped by `git add`. |
| Pre-create stub `.niwa/claude/workspace.md`? | **No**. Empty-file friction outweighs discoverability. |
| Embed audit-trail comment? | **No**. Durable header with transient information (timestamp) is noise; the first commit message already records bootstrap intent. |
| `workspace.Scaffold` new signature, or separate helper? | **Separate helper**: `ScaffoldFromSource(dir, opts)`. Keeps existing callers untouched and isolates the template differences. |

## Assumptions

1. The bootstrap flow stages and commits the scaffolded file (per
   Decision 1 / exploration findings). If the design later flips to
   leaving the file uncommitted, the `.gitkeep` decision still holds
   (the user is expected to `git add` and push, and empty dirs still
   get dropped).
2. `--from`-gated entry means the org is always parseable. If a future
   bootstrap mode adds a no-`--from` path, `ScaffoldFromSource` is not
   called (it requires `Org`).
3. The schema doc URL stays at
   `https://github.com/tsukumogami/niwa/blob/main/docs/guides/workspace-config-sources.md`
   per niwa's `CLAUDE.md` reference to that guide. If it moves, the
   scaffold template's link is the only place that needs updating.
4. The `repos/get` failure is silent in the sense that init still
   succeeds — only the `[groups.<vis>]` line falls back. Future
   work may emit a richer diagnostic if patterns like "consistently
   misclassified" emerge.

## Rejected

- **S-A (no `.gitkeep`)**: rejected because the bootstrap flow stages
  the worktree before user hand-off; empty `.niwa/claude/` silently
  drops on `git add`, breaking the documented content-dir convention
  the moment the user uncomments `[claude.content.workspace]`.
- **S-B (commented-only)**: rejected because it forces every user to
  edit the scaffold before `niwa apply` does anything useful, which
  defeats the "minimal ideal scaffold" goal the user explicitly
  asked for.
- **S-C (stub workspace.md + audit comment)**: rejected because the
  stub file invites empty-file friction without earning its keep, and
  the audit comment encodes information (slug, date) that is either
  redundant with the parent repo URL or actively misleading when the
  file outlives the bootstrap event.
